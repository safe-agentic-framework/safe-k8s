# NIST SP 800-218: Secure Software Development Framework (SSDF) Version 1.1 1.1

- Framework Code: NIST-SSDF
- Version: 1.1
- Total Reverse Mappings: 68
- Distinct Requirements Mapped: 20

## Requirements

- [PO.1.1 - Identify and document all security requirements for the organization's software development infrastructures and processes, and maintain the requirements over time.](po-1-1.md)
- [PO.1.3 - Communicate requirements to all third parties who will provide commercial software components to the organization for reuse.](po-1-3.md)
- [PO.2.1 - Create new roles and alter responsibilities for existing roles as needed to encompass all parts of the SDLC.](po-2-1.md)
- [PO.2.2 - Provide role-based training for all personnel with responsibilities that contribute to secure development.](po-2-2.md)
- [PO.3.2 - Follow recommended security practices to deploy, operate, and maintain tools and toolchains.](po-3-2.md)
- [PO.3.3 - Configure tools to generate artifacts of their support of secure software development practices as defined by the organization.](po-3-3.md)
- [PO.5.1 - Separate and protect each environment involved in software development.](po-5-1.md)
- [PS.1.1 - Store all forms of code -- including source code, executable code, and configuration-as-code -- based on the principle of least privilege.](ps-1-1.md)
- [PS.2.1 - Make software integrity verification information available to software acquirers.](ps-2-1.md)
- [PS.3.1 - Securely archive the necessary files and supporting data to be retained for each software release.](ps-3-1.md)
- [PS.3.2 - Collect, safeguard, maintain, and share provenance data for all components of each software release (e.g., in an SBOM).](ps-3-2.md)
- [PW.1.1 - Use forms of risk modeling -- such as threat modeling, attack modeling, or attack surface mapping -- to help assess the security risk.](pw-1-1.md)
- [PW.4.1 - Acquire and maintain well-secured software components from commercial, open-source, and other third-party developers.](pw-4-1.md)
- [PW.4.4 - Verify that acquired third-party software components comply with requirements throughout their life cycles.](pw-4-4.md)
- [PW.9.1 - Define a secure baseline by determining how to configure each setting that has an effect on security.](pw-9-1.md)
- [PW.9.2 - Implement the default settings (or groups of default settings) and document each setting for software administrators.](pw-9-2.md)
- [RV.1.1 - Gather information from software acquirers, users, and public sources on potential vulnerabilities in the software and third-party components, and investigate all credible reports.](rv-1-1.md)
- [RV.1.3 - Have a policy that addresses vulnerability disclosure and remediation, and implement the roles, responsibilities, and processes needed to support that policy.](rv-1-3.md)
- [RV.2.1 - Analyze each vulnerability to gather sufficient information about risk to plan its remediation or other risk response.](rv-2-1.md)
- [RV.2.2 - Plan and implement risk responses for vulnerabilities.](rv-2-2.md)
