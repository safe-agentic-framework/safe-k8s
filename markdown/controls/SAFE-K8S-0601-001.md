# SAFE-K8S-0601-001 - Base image selection and hardening

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.1 - Container Image and Registry Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

All container images deployed to production clusters must be built from approved minimal base images (distroless, scratch, or organization-approved hardened bases), must use multi-stage builds to exclude build-time tooling, and must remove unnecessary packages. AI workload images must use validated CUDA base images and ML framework images that have been assessed for GPU compatibility and stripped of non-essential components.

## Objective

Minimize the attack surface of container images by enforcing minimal base image selection and hardening practices, with specific attention to the enlarged dependency footprint of AI/ML framework images.

## Rationale

Bloated base images carry unnecessary packages, libraries, and shell utilities that expand the exploitable attack surface. AI workload images are especially prone to this because CUDA toolkits, ML frameworks, and inference servers bundle large native library chains. Without strict base image selection and hardening, adversaries can leverage pre-installed utilities or known CVEs in unused dependencies to achieve code execution, privilege escalation, or lateral movement.

## Scope

Applies to all container images built or deployed in production and pre-production Kubernetes clusters, including inference server images, training job images, and model-serving images. Exceptions require documented justification and compensating controls approved by Security Engineering.

## Enforcement

- Primary Enforcement Point: CI/CD pipeline image build stage and container registry admission policies.
- Implementation Surface: Dockerfile and OCI build configurations (multi-stage builds, USER directive, package removal commands), approved base image registry allowlists (distroless, scratch, nvidia/cuda runtime, tensorflow/tensorflow GPU, nvcr.io/nvidia/tritonserver -min variants), CI/CD pipeline image build stages, container image layer inspection tooling (dive, ldd), CUDA/cuDNN/PyTorch/TensorFlow/vLLM base image catalogs, and internal container registry curated base image repositories.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: ML Platform Engineering; Security Engineering
