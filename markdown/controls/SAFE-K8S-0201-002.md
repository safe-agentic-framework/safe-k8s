# SAFE-K8S-0201-002 - Kubelet certificate rotation and TLS bootstrap

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.1 - Kubelet and Node Configuration Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubelet instances must enable automatic client certificate rotation via TLS bootstrapping and must use valid serving certificates for their HTTPS endpoints, ensuring that expired or self-signed certificates do not remain in use.

## Objective

Maintain continuous cryptographic identity assurance for kubelet-to-API-server communication and kubelet serving endpoints, preventing credential staleness and man-in-the-middle attacks.

## Rationale

Kubelet client certificates authenticate each node to the API server. Without automatic rotation, certificates eventually expire, causing node disconnection or forcing manual renewal that delays patching. Expired or self-signed serving certificates also allow attackers to impersonate kubelets through man-in-the-middle attacks on intra-cluster traffic, potentially intercepting pod specs, secrets, or exec sessions.

## Scope

Applies to all cluster nodes running kubelet, including control-plane nodes where kubelet manages static pods. For managed services that handle certificate lifecycle internally, evidence of provider-managed rotation and TLS enforcement satisfies this control. Environments using external certificate authorities must document their rotation schedule and monitoring.

## Enforcement

- Primary Enforcement Point: Kubelet configuration and cluster certificate authority configuration at provisioning time, with periodic certificate expiry monitoring.
- Implementation Surface: KubeletConfiguration settings (serverTLSBootstrap, rotateCertificates), certificate-approving controller (kubelet-serving-cert-approver), Kubernetes certificates API, and certificate rotation monitoring infrastructure.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; PKI/Certificate Management
