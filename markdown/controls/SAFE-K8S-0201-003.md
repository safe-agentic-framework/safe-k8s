# SAFE-K8S-0201-003 - Node allocatable resources and eviction thresholds

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.1 - Kubelet and Node Configuration Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes nodes must configure system-reserved and kube-reserved resource allocations, and must set eviction thresholds to ensure that kubelet, container runtime, and OS processes retain sufficient CPU, memory, and storage to maintain node stability under workload pressure.

## Objective

Prevent workload resource exhaustion from destabilizing node-level system components, ensuring that kubelet and the container runtime remain responsive even during resource contention.

## Rationale

Without explicit resource reservations, user workloads can consume all available CPU and memory, starving the kubelet and container runtime of resources needed for health checking, log collection, and pod lifecycle management. This can cause cascading node failures, prevent graceful pod eviction, and create denial-of-service conditions that affect all workloads on the node, including security-critical monitoring and logging agents.

## Scope

Applies to all production cluster nodes, with particular attention to nodes hosting GPU workloads or AI training jobs that can exhibit sudden memory spikes. Managed Kubernetes services that expose allocatable resource configuration must have these values tuned for the node instance type. Exemptions require documented risk acceptance.

## Enforcement

- Primary Enforcement Point: Node provisioning templates and kubelet configuration files specifying --system-reserved, --kube-reserved, and --eviction-hard flags.
- Implementation Surface: KubeletConfiguration resource reservation settings (systemReserved, kubeReserved, evictionHard thresholds, enforceNodeAllocatable), GPU driver resource accounting, and node capacity planning tooling.
- Owner Primary: Platform Engineering
- Owner Supporting: Workload Teams; Capacity Planning
