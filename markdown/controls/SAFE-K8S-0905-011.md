# SAFE-K8S-0905-011 - Canary and A/B deployment isolation and rollback safety

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.5 - AI Supply Chain and Model Lifecycle
- Maturity: Advanced
- Classification: ai-specific
- Superseded By: SAFE-K8S-0905-034

## Statement

A/B testing and canary deployment infrastructure must enforce isolation between model versions to prevent a compromised candidate from affecting the stable production version, must maintain traffic splitting integrity, and must implement automatic rollback on error rate or latency degradation.

## Objective

Ensure that multi-version serving infrastructure maintains isolation between model versions and provides safe, automatic rollback capability.

## Rationale

Without isolation between A/B or canary model versions, a compromised candidate model can affect the stable production version. Without rollback safety, a failed deployment cannot be quickly reverted, extending the blast radius of a bad model release.

## Scope

Applies to all multi-version model serving on Kubernetes, including canary deployments, A/B testing, and blue-green deployments. Covers resource isolation between model versions, traffic splitting configuration, and automatic rollback triggers. Excludes promotion gates (SAFE-K8S-0905-005) and approval workflows (SAFE-K8S-0905-010).

## Enforcement

- Primary Enforcement Point: Canary/A-B deployment controller configurations (Argo Rollouts, Flagger, or equivalent).
- Implementation Surface: Canary and A/B deployment controllers (Argo Rollouts, Flagger), Istio VirtualService or Envoy route configuration for traffic splitting, automatic rollback triggers on error rate or latency degradation, and resource isolation between canary and production model versions.
- Owner Primary: ML Platform Engineering
- Owner Supporting: DevOps Engineering; Security Engineering
