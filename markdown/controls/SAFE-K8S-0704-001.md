# SAFE-K8S-0704-001 - Cloud IAM to Kubernetes RBAC mapping

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must establish and maintain a documented mapping between cloud provider IAM roles and Kubernetes RBAC bindings for managed Kubernetes services (EKS, GKE, AKS), must enforce least-privilege cloud IAM roles for Kubernetes access, and must prevent cloud IAM roles from granting broader Kubernetes permissions than intended by the RBAC design.

## Objective

Ensure that cloud IAM and Kubernetes RBAC operate as a coherent access control system, preventing privilege escalation through IAM-to-RBAC mapping gaps.

## Rationale

Managed Kubernetes services bridge cloud IAM and Kubernetes RBAC, creating a complex two-layer access control system. Misalignment between these layers can create privilege escalation paths: a cloud IAM role may grant cluster-admin equivalent access even when Kubernetes RBAC is restrictive, or Kubernetes RBAC may be correctly configured while cloud IAM permits direct API server access that bypasses RBAC. Without explicit mapping documentation and review, these gaps accumulate as teams grow and roles evolve.

## Scope

Applies to all managed Kubernetes services (EKS, GKE, AKS) and any self-managed clusters integrated with cloud IAM for authentication. The mapping must cover all cloud IAM roles that grant any level of Kubernetes cluster access, including roles used by CI/CD pipelines, monitoring systems, and administrative users. The mapping must be reviewed when cloud IAM roles or Kubernetes RBAC bindings are modified.

## Enforcement

- Primary Enforcement Point: Cloud IAM role definitions, Kubernetes RBAC bindings (aws-auth ConfigMap for EKS, GKE IAM bindings, AKS Azure AD integration), and periodic access review processes comparing cloud IAM grants to intended Kubernetes permissions.
- Implementation Surface: Cloud provider IAM-to-Kubernetes RBAC mapping mechanisms (AWS EKS aws-auth ConfigMap and EKS access entries, GCP GKE IAM role bindings and container.clusterAdmin, Azure AKS Entra ID integration and Azure Kubernetes Service Cluster Admin role), system:masters group binding audit and removal, break-glass account procedures with multi-person authorization, cloud IAM analysis tools (AWS IAM Access Analyzer, GCP IAM Recommender, Azure AD Access Reviews), and quarterly IAM-to-RBAC mapping review documentation.
- Owner Primary: Cloud Platform Engineering
- Owner Supporting: Security Engineering; Platform Engineering
