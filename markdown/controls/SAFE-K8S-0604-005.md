# SAFE-K8S-0604-005 - Dependency tracking and SLSA build provenance

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.4 - SBOMs and Vulnerability Intelligence
- Maturity: Practitioner
- Classification: baseline

## Statement

Build pipelines must implement SLSA provenance generation at the target SLSA level defined by organizational policy, use hermetic builds that prevent uncontrolled network access and dependency resolution during build execution, and produce verifiable source-to-artifact integrity chains that link each deployed image to its exact source commit, dependency lock files, and build configuration.

## Objective

Establish auditable, tamper-evident traceability from deployed artifacts back to their source code and dependencies, preventing dependency confusion, typosquatting, and build-time injection attacks.

## Rationale

Non-hermetic builds resolve dependencies at build time from external sources, creating a window for dependency confusion and typosquatting attacks. Without SLSA provenance, there is no verifiable link between a deployed image and its source, making it impossible to determine whether an artifact was built from authorized code. Build-time injection attacks exploit this gap to insert malicious code that passes all post-build scans because it was never in the source repository.

## Scope

Applies to all CI/CD pipelines producing deployable artifacts for Kubernetes clusters. Includes application images, AI workload images, and infrastructure operator images. The target SLSA level must be documented and reviewed annually, with a roadmap for progressive improvement.

## Enforcement

- Primary Enforcement Point: CI/CD pipeline build configuration and provenance verification at admission time.
- Implementation Surface: SLSA provenance generation tooling (slsa-github-generator, Google Cloud Build native SLSA Level 3), hermetic build environment configuration (no outbound network access, pre-resolved dependency mirrors), dependency lock file integrity verification (pip install --require-hashes, go.sum, package-lock.json), dependency proxy/mirror caches for approved packages, source-to-artifact integrity chain verification at admission, and CI/CD build configuration enforcing pinned dependencies with hash verification.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: Security Engineering; Platform Engineering
