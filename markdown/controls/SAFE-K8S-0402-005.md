# SAFE-K8S-0402-005 - Projected service account token configuration

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.2 - Service Accounts and Workload Identity
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes clusters must use projected service account tokens with audience-scoped and time-bound configurations.

## Objective

Replace legacy non-expiring service account tokens with projected tokens that are automatically rotated, scoped to specific audiences, and time-limited, reducing the window and scope of token compromise.

## Rationale

Projected service account tokens are bound to a specific audience and expiration time, automatically rotated by the kubelet, and validated by the API server against the token request record, dramatically reducing the impact of token exfiltration. Unlike legacy static tokens, projected tokens cannot provide indefinite cluster access after theft.

## Scope

Applies to all clusters running Kubernetes 1.22 or later. All workloads must use projected volume-mounted tokens or TokenRequest API tokens with explicit audience and expiration configuration.

## Enforcement

- Primary Enforcement Point: Admission policies requiring projected token volumes with explicit audience and expirationSeconds fields, and verification of BoundServiceAccountTokenVolume feature gate configuration.
- Implementation Surface: Projected ServiceAccountToken volume configurations (expirationSeconds, audience fields), TokenRequest API, admission policies enforcing projected token usage, and BoundServiceAccountTokenVolume feature gate verification.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development
