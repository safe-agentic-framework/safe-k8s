# SAFE-K8S-0704-010 - Privileged cloud-to-cluster access boundary enforcement and break-glass control

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must enforce least-privilege cloud IAM roles for Kubernetes access, must prevent cloud IAM roles from granting broader Kubernetes permissions than intended by the RBAC design, and must restrict cluster-admin-equivalent or break-glass cloud identities to tightly governed emergency access procedures.

## Objective

Prevent privilege escalation through cloud IAM identities that resolve to excessive Kubernetes permissions or uncontrolled administrative access paths.

## Rationale

Even with a documented IAM-to-RBAC map, security fails if cloud identities still resolve to broad Kubernetes privileges or if break-glass access remains permanently available. Restricting cluster-admin-equivalent paths and requiring explicit emergency access governance closes the gap between documented intent and actual effective privilege.

## Scope

Applies to managed Kubernetes services and any cloud-authenticated administrative or automation identities that can obtain privileged Kubernetes access. Includes break-glass accounts, vendor-admin paths, and any identity mapped to system:masters, cluster-admin, or equivalent broad authorization.

## Enforcement

- Primary Enforcement Point: Cloud IAM role design, privileged access controls, and emergency access governance for Kubernetes administration.
- Implementation Surface: Cloud IAM least-privilege role definitions, audits of system:masters or cluster-admin-equivalent mappings, break-glass account procedures, multi-person approval for emergency access, and access review evidence showing remediation of privileged mapping drift.
- Owner Primary: Cloud Platform Engineering
- Owner Supporting: Security Engineering; Platform Engineering
