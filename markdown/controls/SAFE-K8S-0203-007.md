# SAFE-K8S-0203-007 - Pod metadata endpoint blocking and workload identity replacement

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.3 - Host OS and Kernel Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Cloud-hosted Kubernetes clusters must implement network-level controls that prevent pod workloads from directly accessing cloud metadata endpoints, and must replace metadata-derived cloud credentials with workload identity federation where the platform supports it.

## Objective

Block workload paths to cloud metadata endpoints and eliminate the need for metadata-derived credentials by shifting workloads to dedicated workload identities.

## Rationale

Even when provider metadata hardening is enabled, pod workloads may still attempt to reach metadata endpoints through host networking or local routing paths. Network-level blocking constrains pod reachability, and workload identity federation removes the operational need for pods to access node instance metadata for cloud API credentials.

## Scope

Applies to cloud-hosted Kubernetes clusters where pods can otherwise route to metadata endpoints such as 169.254.169.254. Covers NetworkPolicy, host firewall, iptables, CNI controls, and workload identity federation features such as IRSA, EKS Pod Identity, GKE Workload Identity, and Azure Workload Identity.

## Enforcement

- Primary Enforcement Point: Node and cluster networking controls plus workload identity configuration for pod credential access paths.
- Implementation Surface: NetworkPolicy or host-level filtering for metadata endpoints, validation tests confirming pods cannot reach metadata services, and workload identity federation configuration replacing node instance credentials for cloud API access.
- Owner Primary: Cloud Infrastructure
- Owner Supporting: Platform Engineering; Security Engineering
