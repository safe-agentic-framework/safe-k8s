# SAFE-K8S-0202-001 - Container runtime configuration baselines and socket protection

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.2 - Container Runtime Security
- Maturity: Foundational
- Classification: baseline

## Statement

Container runtimes (containerd or CRI-O) must be configured according to an approved security baseline that restricts runtime socket access to root-only permissions, enforces secure default runtime settings, and disables unnecessary features such as deprecated API versions or debug endpoints.

## Objective

Establish a hardened container runtime configuration that minimizes the runtime attack surface and prevents unauthorized access to the runtime control socket.

## Rationale

The container runtime socket (e.g., /run/containerd/containerd.sock) provides full control over container lifecycle operations including creation, exec, and image pulling. Permissive socket access allows any process with socket connectivity to create privileged containers, escape to the host, or exfiltrate data from other containers. Insecure default runtime settings such as enabled debug handlers or unrestricted image registries expand the attack surface unnecessarily.

## Scope

Applies to all nodes running containerd or CRI-O in production clusters. Managed Kubernetes services where the runtime is provider-managed satisfy this control through provider security documentation and compensating network controls. Custom or alternative CRI implementations must demonstrate equivalent hardening.

## Enforcement

- Primary Enforcement Point: Node image build pipeline and container runtime configuration files (config.toml for containerd, crio.conf for CRI-O).
- Implementation Surface: Container runtime configuration files (containerd config.toml, CRI-O crio.conf), runtime socket permissions and ownership settings, debug endpoint configuration, and runtime configuration verification commands.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; OS/Infrastructure
