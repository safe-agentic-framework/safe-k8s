# SAFE-K8S-0101-006 - API server audit logging policy and backend configuration

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.1 - Kubernetes API Server Security
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must enable API server audit logging with an audit policy that captures at least Metadata-level events for all resource types, Request-level events for sensitive resources (Secrets, ConfigMaps, RBAC bindings, TokenReviews), and RequestResponse-level events for authentication and authorization decisions. Audit logs must be directed to a durable backend (log file with rotation, or webhook to a SIEM) and retained for a minimum period defined by organizational policy.

## Objective

Ensure that all security-relevant API server activity is recorded in a tamper-evident audit trail that supports incident investigation, compliance evidence, and anomaly detection.

## Rationale

API server audit logs are the authoritative record of who did what in a Kubernetes cluster. Without a well-defined audit policy, critical events may be omitted or logged at insufficient detail, undermining forensic capability. Without a durable backend, logs may be lost on node restart or disk pressure. Audit logging was previously combined with general API server hardening but warrants its own control due to the distinct configuration surface (audit policy file, backend flags, retention settings) and the separate expertise required to tune policy levels without overwhelming log volume.

## Scope

Applies to all production Kubernetes clusters. For managed Kubernetes services, verify the provider's audit logging defaults and configure customer-accessible audit policy and log export settings. Covers kube-apiserver --audit-policy-file, --audit-log-path, --audit-log-maxage, --audit-log-maxbackup, --audit-log-maxsize, and --audit-webhook-config-file flags.

## Enforcement

- Primary Enforcement Point: API Server Configuration
- Implementation Surface: API server static pod manifest or managed service audit configuration, audit policy YAML file, log rotation configuration, SIEM or log aggregation webhook endpoint.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Compliance
