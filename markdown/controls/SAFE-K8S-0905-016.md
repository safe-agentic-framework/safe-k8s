# SAFE-K8S-0905-016 - Model registry RBAC, service account scoping, and access review

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.5 - AI Supply Chain and Model Lifecycle
- Maturity: Practitioner
- Classification: ai-specific
- Superseded By: SAFE-K8S-0905-038

## Statement

Model registries must enforce role-based access controls for publish, pull, modify, and delete operations, scope workload and service-account access to approved repositories and environments, and perform periodic access reviews to detect excessive or stale registry permissions.

## Objective

Prevent unauthorized publication, retrieval, and modification of model artifacts by tightly controlling which users, groups, and workloads can access specific model repositories and versions.

## Rationale

The model registry is the operational source of truth for deployable model artifacts. If registry privileges are broad or unreviewed, an attacker or over-privileged workload can publish malicious models, pull restricted model weights, or bypass environment approvals.

## Scope

Applies to all model registry deployments on Kubernetes, including Harbor, MLflow Model Registry, and on-premises Hugging Face Hub instances. Covers project-scoped roles, OIDC or IAM integration, workload pull permissions, and periodic review of registry access. Excludes replication-channel security and model-retention lifecycle governance.

## Enforcement

- Primary Enforcement Point: Model registry RBAC and IAM configuration, service-account authorization policy, and periodic access review workflow.
- Implementation Surface: Project-scoped registry roles, OIDC and cloud IAM integration, registry robot or service-account credentials, per-environment repository access rules, pull-only permissions for inference workloads, and access-review evidence for human and workload identities.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Security Engineering; Platform Engineering
