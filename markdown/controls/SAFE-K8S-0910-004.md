# SAFE-K8S-0910-004 - Federated learning cross-cluster coordination security

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.10 - Multi-Cluster and Federated AI Security
- Maturity: Advanced
- Classification: ai-specific

## Statement

Federated learning deployments that span multiple Kubernetes clusters must implement cross-cluster participant authentication, secure aggregation that operates across cluster trust boundaries, consistent differential privacy enforcement regardless of participant cluster location, and coordination protocols that are resilient to the compromise of any single participating cluster.

## Objective

Extend federated learning security controls to cross-cluster deployments where participants run in separate clusters with distinct trust domains, ensuring that cross-cluster FL provides equivalent security guarantees to single-cluster FL while addressing additional cross-boundary threats.

## Rationale

Single-cluster federated learning (SAFE-K8S-0901-005) assumes participants are isolated by namespace-level controls within one cluster. Cross-cluster FL introduces additional threats: compromised clusters submitting poisoned gradients through legitimate federation channels, gradient interception at cluster boundaries, inconsistent privacy controls across heterogeneous environments, and the inability to enforce namespace-level isolation across cluster boundaries. Without dedicated cross-cluster FL controls, the security guarantees of single-cluster FL break down at federation boundaries.

## Scope

Applies to federated learning deployments where participants run in separate Kubernetes clusters. Covers cross-silo FL across organizational clusters and cross-cloud FL across cloud provider boundaries. Single-cluster FL security is addressed by SAFE-K8S-0901-005.

## Enforcement

- Primary Enforcement Point: Cross-cluster FL coordination service configuration, privacy budget coordinator deployment, cross-cluster participant authentication settings, and Byzantine-resilient aggregation threshold configuration.
- Implementation Surface: Cross-cluster FL coordination service configuration, cryptographic identity federation (SPIFFE/SPIRE) for participant authentication bound to cluster of origin, cross-boundary secure aggregation protocols (hierarchical local and global aggregation), centralized privacy budget coordinator tracking cumulative epsilon spend across all clusters, Byzantine-resilient aggregation (Krum, coordinate-wise median) with threshold tolerating at least one compromised cluster, contribution anomaly detection per cluster, and centralized cross-cluster FL audit logging outside any single participant cluster.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Security Engineering; Privacy Engineering; Platform Engineering
