# SAFE-K8S-0104-003 - Kubernetes CVE monitoring, prioritization, and emergency patching

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.4 - CIS Benchmarks and Patch Management
- Maturity: Foundational
- Classification: baseline

## Statement

Organizations must monitor Kubernetes security announcements (kubernetes-security-announce mailing list, CVE databases, and provider advisories), prioritize CVEs based on severity, exploitability, and cluster exposure, and maintain documented emergency upgrade procedures that enable critical patches to be applied within organization-defined SLA windows.

## Objective

Reduce the window of exposure to known Kubernetes vulnerabilities by establishing a structured CVE response process that covers detection, prioritization, and rapid patching.

## Rationale

Kubernetes CVEs are publicly disclosed with increasing frequency, and exploit code often follows within days. Without active monitoring, security teams may be unaware of critical vulnerabilities affecting their clusters. Without a prioritization framework, all CVEs receive equal treatment, delaying response to the most critical issues. Without pre-established emergency upgrade procedures, patch deployment under time pressure introduces operational risk and delays.

## Scope

Applies to all production and non-production clusters. Covers Kubernetes core component CVEs, container runtime CVEs, and critical add-on vulnerabilities (CNI, CSI, ingress controllers). For managed Kubernetes services, applies to monitoring provider patch timelines, enabling automatic security patching where available, and coordinating shared responsibility for component-level CVEs. Also covers lifecycle management of cluster add-ons with independent release cycles (CoreDNS, CNI plugins, CSI drivers, ingress controllers, cert-manager).

## Enforcement

- Primary Enforcement Point: Governance / Process
- Implementation Surface: Security announcement monitoring integration (mailing list subscriptions, RSS feeds, vulnerability scanner feeds), CVE prioritization workflow, and emergency change management procedures.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Change Management; Incident Response
