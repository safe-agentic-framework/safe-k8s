# SAFE-K8S-0801-003 - MPS and time-slicing isolation and memory safety controls

- Domain: D08 - GPU, Accelerator, and Confidential Computing
- Knowledge Area: 8.1 - GPU Device Plugins and Resource Allocation
- Maturity: Advanced
- Classification: ai-specific

## Statement

Clusters using MPS (Multi-Process Service) or GPU time-slicing must document the reduced isolation guarantees compared to MIG, implement GPU memory isolation verification, perform side-channel resistance assessments, and enforce cross-tenant data remnant prevention measures including GPU memory clearing between workload transitions.

## Objective

Ensure that organizations consciously accept and mitigate the weaker isolation properties of MPS and time-slicing compared to hardware-partitioned approaches, with specific controls to prevent GPU memory leakage and side-channel attacks between co-located workloads.

## Rationale

MPS and time-slicing share GPU compute and memory resources without hardware isolation barriers, meaning one workload can potentially read another's GPU memory contents or infer data through timing side channels. GPU memory is not automatically zeroed between process context switches in time-sliced mode, leaving training data, model weights, or inference inputs recoverable by subsequent tenants. Without explicit risk acceptance and compensating controls, organizations may unknowingly expose sensitive AI workloads to cross-tenant data leakage.

## Scope

Applies to any Kubernetes cluster using NVIDIA MPS or time-slicing for GPU sharing. Workloads processing sensitive data (PII, proprietary models, regulated data) must not use time-slicing without documented risk acceptance and compensating controls. MIG-capable hardware should prefer MIG over time-slicing for multi-tenant isolation.

## Enforcement

- Primary Enforcement Point: GPU Operator time-slicing configuration, MPS daemon settings, admission control policies restricting time-sliced GPU access for sensitive workloads, and periodic GPU memory isolation verification tests.
- Implementation Surface: GPU Operator time-slicing configuration, NVIDIA MPS daemon settings and environment variables (CUDA_MPS_ACTIVE_THREAD_PERCENTAGE, CUDA_MPS_PINNED_DEVICE_MEM_LIMIT), admission control policies restricting time-sliced/MPS GPU access for sensitive workloads, node affinity and taint/toleration rules for same-trust co-location, GPU memory clearing mechanisms (cudaMemset, nvidia-smi -r GPU reset), risk register documentation for MPS/time-slicing risk acceptance, and side-channel assessment process integrated with workload classification labels.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Security Engineering; Data Protection Office
