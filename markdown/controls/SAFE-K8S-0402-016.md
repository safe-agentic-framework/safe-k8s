# SAFE-K8S-0402-016 - Projected service account token audience and TokenRequest enforcement

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.2 - Service Accounts and Workload Identity
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes clusters must issue workload service account tokens through projected service account token volumes or the TokenRequest API with explicit audience configuration, and must enforce these issuance paths through admission and policy controls instead of relying on legacy secret-mounted tokens.

## Objective

Ensure workload tokens are issued through explicit, audience-scoped projected token mechanisms rather than implicit legacy secret mounting.

## Rationale

Projected service account token volumes and TokenRequest API issuance let clusters bind workload tokens to explicit audiences and validated token request flows. This reduces the chance that a stolen token can be replayed against unintended services and provides a verifiable issuance path for workload identity.

## Scope

Applies to all clusters running projected service account tokens for workload authentication. Workload tokens must be issued through projected volumes or TokenRequest flows with explicit audience configuration appropriate to the target API or external service.

## Enforcement

- Primary Enforcement Point: Admission and policy enforcement for workload token issuance method and audience configuration.
- Implementation Surface: Projected ServiceAccountToken volume configuration, TokenRequest API usage, admission policies enforcing projected-token issuance paths, and validation of explicit audience settings for workload tokens used against the API server or external relying services.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development
