# SAFE-K8S-0102-011 - etcd certificate validity limits and expiration alerting

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters must enforce organization-defined maximum validity periods for etcd server, peer, and API server-to-etcd client certificates, and must generate monitoring alerts with sufficient lead time to remediate certificates approaching expiration.

## Objective

Ensure that etcd certificates remain short-lived enough to reduce key-compromise exposure and visible enough to prevent outages from unnoticed expiration.

## Rationale

Automated rotation is insufficient when certificate validity windows are overly long or expiration is not actively monitored. This control isolates certificate policy and alerting so operators can independently audit TTL constraints, expiration dashboards, and notification thresholds.

## Scope

Applies to all self-managed Kubernetes clusters using etcd. Covers certificate lifetime policy, issuance profiles, expiration telemetry, and alert routing for etcd-related certificates.

## Enforcement

- Primary Enforcement Point: Certificate Policy / Monitoring
- Implementation Surface: CA or issuer policy configuration, certificate validity settings, expiration dashboards, alert rules, and notification routing for etcd certificate expiry.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
