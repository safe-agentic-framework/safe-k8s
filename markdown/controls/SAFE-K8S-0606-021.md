# SAFE-K8S-0606-021 - Kubernetes manifest cryptographic signing before deployment

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.6 - CI/CD and GitOps Pipeline Security
- Maturity: Advanced
- Classification: baseline

## Statement

Kubernetes deployment manifests must be cryptographically signed before promotion or deployment, and the signing step must be executed in the CI/CD workflow using authorized signing identities and protected signing configuration.

## Objective

Ensure deployment manifests become tamper-evident artifacts before they leave the delivery workflow.

## Rationale

If manifests are not signed before promotion, an attacker who modifies deployment configuration after image signing can alter resource limits, security contexts, environment variables, or image references without a cryptographic trace. Manifest signing is therefore a distinct authoring and promotion control even when later admission checks also exist.

## Scope

Applies to Kubernetes deployment manifests used in staging and production delivery workflows. Development-only environments may use reduced signing requirements only with documented justification.

## Enforcement

- Primary Enforcement Point: CI/CD manifest signing step before promotion or deployment.
- Implementation Surface: Manifest signing configuration, authorized signing identities, CI/CD signing workflow steps, and signed manifest artifacts produced before deployment.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: Platform Engineering; Security Engineering
