# SAFE-K8S-0606-005 - Helm chart security and provenance

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.6 - CI/CD and GitOps Pipeline Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Helm charts used for Kubernetes deployments must have verified provenance (helm verify), chart signatures must be validated before installation, values injection must be controlled to prevent override of security-critical settings, and all chart dependencies must be pinned to specific versions with integrity verification. Charts from external sources must be sourced from trusted repositories and validated before use.

## Objective

Prevent supply chain attacks through Helm charts by verifying chart provenance, controlling values injection, and pinning dependencies to known-good versions.

## Rationale

Helm charts define Kubernetes resource manifests and can embed arbitrary templates, hooks, and sub-charts. A compromised chart can create privileged pods, exfiltrate secrets, or modify cluster configuration. Without provenance verification, organizations cannot distinguish legitimate charts from tampered ones. Unpinned dependencies allow silent substitution of malicious sub-charts. Uncontrolled values injection enables attackers to override security contexts, resource limits, or image references through values files.

## Scope

Applies to all Helm chart usage in production and staging Kubernetes clusters, including charts for application deployments, infrastructure operators, and AI workload platforms (KServe, Kubeflow, Ray). Covers both internally developed and externally sourced charts.

## Enforcement

- Primary Enforcement Point: Helm chart repository configuration, CI/CD pipeline chart validation step, and GitOps reconciliation chart verification.
- Implementation Surface: Helm provenance files (.prov) and helm verify CLI for chart signature validation, Chart.lock dependency pinning with digest verification (helm dependency build --verify), values.yaml schema validation preventing override of security-critical settings (securityContext, serviceAccount, hostNetwork), internal Helm chart repository mirroring for external charts after security review, and CI/CD pipeline chart validation steps verifying signatures before GitOps reconciliation or direct deployment.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: Platform Engineering; Security Engineering
