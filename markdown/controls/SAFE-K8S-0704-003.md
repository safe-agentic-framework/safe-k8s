# SAFE-K8S-0704-003 - Cloud audit log correlation with Kubernetes audit events

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must integrate cloud provider audit logs (AWS CloudTrail, GCP Cloud Audit Logs, Azure Activity Log) with Kubernetes audit events into a unified security monitoring platform, and must implement correlation rules that link cloud-level actions to their Kubernetes-level effects to enable end-to-end incident investigation.

## Objective

Enable unified security monitoring and incident investigation by correlating cloud infrastructure events with Kubernetes control plane activity.

## Rationale

Security incidents in managed Kubernetes environments span both cloud and Kubernetes layers. An attacker who modifies a cloud IAM role to gain Kubernetes access generates events in both cloud audit logs and Kubernetes audit logs. Without correlation, security teams investigating an incident see only half the picture and cannot trace the full attack chain. Disconnected audit streams also create blind spots where attacker activity in one layer is not correlated with suspicious activity in the other.

## Scope

Applies to all managed Kubernetes clusters on cloud providers (EKS, GKE, AKS). Cloud audit logs must be forwarded to the same SIEM or log analytics platform as Kubernetes audit events. Correlation rules must cover at minimum: IAM role changes affecting Kubernetes access, node provisioning events, load balancer modifications, and storage operations. Log retention must meet organizational and regulatory requirements.

## Enforcement

- Primary Enforcement Point: Cloud audit log forwarding configuration, Kubernetes audit policy and log forwarding, SIEM correlation rules, and log retention policies.
- Implementation Surface: Cloud provider audit log services (AWS CloudTrail, GCP Cloud Audit Logs, Azure Activity Log), Kubernetes audit log events, unified SIEM or log analysis platform (Splunk, Elastic, Google SecOps), correlation rules linking cloud IAM identity to Kubernetes user identity via IAM-to-RBAC mapping, cloud request ID to Kubernetes audit request metadata correlation, timestamp-based event correlation windows, and detection rules for high-priority scenarios (IAM role assumption followed by cluster-admin actions, node pool modifications correlated with scheduling changes, VPC flow log metadata service access correlated with pod identity).
- Owner Primary: Security Engineering
- Owner Supporting: Cloud Platform Engineering; Platform Engineering; SOC
