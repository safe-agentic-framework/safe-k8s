# SAFE-K8S-0606-008 - GitOps reconciliation integrity and drift detection

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.6 - CI/CD and GitOps Pipeline Security
- Maturity: Practitioner
- Classification: baseline

## Statement

GitOps controllers (Flux or ArgoCD) must verify reconciliation integrity and detect runtime configuration deviations from Git-declared state with automated resync or notification. Manual runtime changes (kubectl apply) must be prohibited in production through cluster RBAC policies, with Git as the single source of truth for all deployed resources.

## Objective

Prevent unauthorized runtime modifications to cluster state and detect configuration drift by enforcing Git as the sole deployment path and monitoring for deviations.

## Rationale

Without drift detection, runtime modifications made via kubectl or direct API access persist undetected, creating a divergence between the declared and actual cluster state that undermines auditability and incident response. Prohibiting manual runtime changes through RBAC forces all modifications through the Git-based review and audit trail. Automated resync ensures detected drift is corrected without manual intervention.

## Scope

Applies to all production Kubernetes clusters managed through GitOps (Flux or ArgoCD). Covers reconciliation configuration, drift detection, and cluster RBAC policies restricting direct API modifications. Non-production clusters used for experimentation may allow direct kubectl access with documented justification.

## Enforcement

- Primary Enforcement Point: Flux/ArgoCD reconciliation configuration and cluster RBAC restricting direct API modifications.
- Implementation Surface: ArgoCD auto-sync with selfHeal: true and Flux reconcile interval for drift detection and automated resync, cluster RBAC policies denying create/update/delete on Deployments, StatefulSets, and CRDs for human users in production namespaces, drift detection alert notifications for manual runtime changes, and audit logs capturing any emergency direct access with break-glass justification.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; SRE / Operations
