# SAFE-K8S-0604-006 - Vulnerability prioritization for AI workloads

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.4 - SBOMs and Vulnerability Intelligence
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Vulnerability management processes must distinguish training-time from inference-time exposure, separately prioritize GPU driver CVEs from application-layer CVEs based on workload context, and maintain automated rebuild and redeployment pipelines that can patch AI workload images without requiring model retraining. Vulnerability SLAs must reflect the criticality and exposure profile of each AI workload tier.

## Objective

Ensure AI workload vulnerabilities are prioritized based on their actual exploitation context rather than generic severity scores, and that patching does not require expensive model retraining cycles.

## Rationale

Generic CVSS-based prioritization does not account for AI workload context. A GPU driver CVE in a training cluster with no external network exposure poses different risk than the same CVE in an internet-facing inference endpoint. Without context-aware prioritization, critical inference-tier vulnerabilities may be deprioritized while low-risk training-only findings consume remediation bandwidth. Without automated rebuild pipelines, patching requires retraining, creating perverse incentives to defer patching.

## Scope

Applies to all AI/ML workload images deployed in Kubernetes clusters, including inference servers, training jobs, data preprocessing pipelines, and model-serving infrastructure. Covers both application-layer dependencies and GPU/CUDA driver stack vulnerabilities.

## Enforcement

- Primary Enforcement Point: Vulnerability management platform policy configuration and CI/CD automated rebuild pipeline triggers.
- Implementation Surface: Vulnerability management platform policy configuration distinguishing training-time vs. inference-time exposure, GPU driver CVE vs. application-layer CVE prioritization rules, workload-tier-specific vulnerability SLA definitions, automated CI/CD rebuild and redeployment pipelines triggered by base image or dependency patches, model weight loading at runtime from model registries (not baked into container images), and regression test automation for patched AI workload images.
- Owner Primary: Security Engineering
- Owner Supporting: ML Platform Engineering; DevSecOps / Release Engineering
