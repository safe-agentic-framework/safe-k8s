# SAFE-K8S-0604-012 - Hermetic build dependency control and source-to-artifact integrity

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.4 - SBOMs and Vulnerability Intelligence
- Maturity: Practitioner
- Classification: baseline

## Statement

Build pipelines must use hermetic build configurations that prevent uncontrolled network access or live dependency resolution during build execution, must enforce committed dependency lock files and pinned dependency inputs, and must maintain source-to-artifact integrity links tying each deployed image to its exact source commit, dependency lock files, and build configuration.

## Objective

Prevent dependency confusion, typosquatting, and build-time injection by ensuring builds run from approved, pre-resolved inputs and preserve an auditable chain from source and dependency state to the produced artifact.

## Rationale

Even when provenance is generated, non-hermetic builds that resolve packages dynamically can still ingest malicious or unexpected content at build time. Hermetic execution, lockfile enforcement, and source-to-artifact integrity collectively reduce that attack surface and make dependency state reproducible for review.

## Scope

Applies to all CI/CD pipelines producing deployable Kubernetes artifacts, including application images, AI workload images, and infrastructure operator images. Approved mirrors, proxies, or caches may be used when they preserve pinned dependency and integrity guarantees.

## Enforcement

- Primary Enforcement Point: CI/CD build environment configuration and dependency resolution controls.
- Implementation Surface: Hermetic build settings, no-egress build workers, approved dependency mirrors, committed lock files with hash validation, and traceable linkage from source commit and dependency state to the built artifact.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: Security Engineering; Platform Engineering
