# SAFE-K8S-0303-002 - AppArmor profile management and enforcement

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.3 - Mandatory Access Controls
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters using AppArmor-enabled nodes must load approved AppArmor profiles on all nodes via node provisioning automation, annotate pods with the appropriate AppArmor profile, and must not permit containers to run in unconfined mode unless a documented exception with compensating controls exists.

## Objective

Enforce mandatory access control at the application level through AppArmor profiles that constrain each container's file access, network operations, and capability usage beyond what seccomp and security contexts alone provide.

## Rationale

AppArmor provides path-based mandatory access control that limits which files and directories a container can read, write, or execute, and which network operations it can perform. Without AppArmor profiles, a compromised container can access any file the container user has permission to reach, including sensitive mounted volumes and device files. Inconsistent profile loading across nodes creates enforcement gaps that attackers can exploit through pod scheduling manipulation.

## Scope

Applies to all clusters running on nodes with AppArmor support (Ubuntu, Debian, SUSE, and other distributions using AppArmor as the default LSM). Does not apply to nodes using SELinux as the primary LSM. Profile loading must be consistent across all nodes in a node pool.

## Enforcement

- Primary Enforcement Point: Node provisioning automation (DaemonSets, machine images, cloud-init) for profile loading; pod annotations (container.apparmor.security.beta.kubernetes.io) or securityContext.appArmorProfile for profile assignment; policy engine rules validating profile annotations at admission.
- Implementation Surface: AppArmor profile files loaded on nodes via DaemonSets, cloud-init, Ansible, or machine image baking, pod annotations (container.apparmor.security.beta.kubernetes.io) or securityContext.appArmorProfile (Kubernetes 1.30+), policy engine mutation rules injecting runtime/default for pods without AppArmor configuration, node provisioning automation for consistent profile distribution, aa-logprof and complain-mode testing tooling, and node-level AppArmor verification (sys/module/apparmor/parameters/enabled).
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
