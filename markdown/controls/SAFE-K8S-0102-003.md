# SAFE-K8S-0102-003 - etcd backup and restore security

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must encrypt etcd backups at rest and in transit, verify backup integrity before storage and before restore operations, and restrict access to backup storage to authorized personnel and automated processes only.

## Objective

Ensure that etcd backups do not become an unprotected copy of the full cluster state by enforcing encryption, integrity verification, and access control throughout the backup and restore lifecycle.

## Rationale

etcd backups contain the complete cluster state, including all Secrets in their stored form. An unencrypted or improperly secured backup provides an attacker with a full copy of cluster data without needing cluster access. Without integrity verification, a corrupted or tampered backup could introduce malicious state during a restore operation. Unrestricted backup storage access expands the attack surface beyond the cluster itself.

## Scope

Applies to all production clusters. Covers etcd snapshot creation, transmission to backup storage, storage-at-rest encryption, integrity validation (checksums or signatures), access controls on backup repositories, and restore procedure security. For managed Kubernetes services, applies to verifying provider backup encryption and access controls and to any customer-managed backup processes.

## Enforcement

- Primary Enforcement Point: Node / OS
- Implementation Surface: Backup automation tooling configuration, backup storage access policies (IAM, bucket policies), and restore procedure runbooks with integrity verification steps.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Backup and Recovery Operations
