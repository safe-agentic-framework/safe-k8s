# SAFE-K8S-0403-001 - Kubernetes Secrets external KMS integration and key lifecycle

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.3 - Secrets Management
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes Secrets encryption at rest must use an external KMS provider (AWS KMS, GCP KMS, Azure Key Vault, HashiCorp Vault, or equivalent) as the key management backend, with automated key rotation, access-controlled key policies, and periodic re-encryption verification to confirm no Secrets remain under retired keys.

## Objective

Ensure that the cryptographic keys protecting Kubernetes Secrets are managed outside the cluster trust boundary with proper lifecycle controls, complementing the API-resource encryption mechanism established by SAFE-K8S-0101-003.

## Rationale

SAFE-K8S-0101-003 establishes that Secrets must be encrypted at rest via EncryptionConfiguration. This control specifies the key management requirements: keys must be managed by an external KMS provider rather than stored locally on the API server, key access policies must follow least privilege, and keys must be rotated on a defined schedule with re-encryption of existing Secrets to retire old key material. Without external KMS, encryption keys are stored alongside the data they protect, negating much of the defense-in-depth benefit. Without key rotation, long-lived keys accumulate risk from potential exposure.

## Scope

Applies to all production Kubernetes clusters. Covers KMS provider selection and integration, key access policy configuration, key rotation scheduling, and re-encryption verification. For managed Kubernetes services, covers customer-managed key (CMK) configuration and provider key management documentation. The underlying encryption mechanism (EncryptionConfiguration, provider selection, KMS v2 preference) is governed by SAFE-K8S-0101-003; this control governs the key management lifecycle.

## Enforcement

- Primary Enforcement Point: KMS provider key policies, key rotation automation, and periodic re-encryption verification confirming all Secrets are encrypted under current keys.
- Implementation Surface: Kubernetes EncryptionConfiguration with KMS provider plugins (AWS KMS, GCP KMS, Azure Key Vault, HashiCorp Vault Transit), KMS v2 API, external KMS key access policies (least-privilege Encrypt/Decrypt permissions), automated key rotation schedules (cloud-native KMS rotation), re-encryption verification tooling (kubectl get/replace for Secret re-encryption), and customer-managed key (CMK) configurations for managed Kubernetes services.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Key Management/Cryptography; Cloud Platform Engineering
