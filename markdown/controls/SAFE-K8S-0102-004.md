# SAFE-K8S-0102-004 - etcd cluster health monitoring, compaction, and defragmentation

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must implement continuous etcd health monitoring with alerts for cluster quorum loss, storage quota exhaustion, and replication lag, and must enforce automated compaction and periodic defragmentation to prevent storage-related denial of service.

## Objective

Maintain etcd availability and performance as security-critical operational requirements by detecting health degradation and preventing storage exhaustion that could cause control-plane outages.

## Rationale

An unhealthy etcd cluster directly impacts Kubernetes control-plane availability. Uncompacted revision history and fragmented storage can cause etcd to exceed its storage quota, resulting in the cluster entering read-only mode and rejecting all write operations, including security- critical actions such as certificate rotation, policy updates, and incident response. Monitoring gaps delay detection of etcd degradation, turning operational issues into security incidents.

## Scope

Applies to all production clusters running self-managed etcd. Covers etcd health endpoint monitoring, quorum status, storage quota tracking, compaction scheduling, and defragmentation operations. For managed Kubernetes services, applies to monitoring provider-exposed etcd health metrics and alerting on control-plane availability degradation.

## Enforcement

- Primary Enforcement Point: Audit / Telemetry
- Implementation Surface: Monitoring and alerting infrastructure (Prometheus, cloud-native monitoring), etcd compaction configuration (--auto-compaction-retention), and scheduled defragmentation automation.
- Owner Primary: Platform Engineering
- Owner Supporting: Site Reliability Engineering; Security Operations
