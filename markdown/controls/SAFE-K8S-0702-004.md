# SAFE-K8S-0702-004 - Virtual cluster isolation for multi-tenant environments

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.2 - Namespace Isolation and Multi-Tenancy
- Maturity: Advanced
- Classification: baseline

## Statement

Organizations requiring stronger tenant isolation than namespace-level controls provide must evaluate and implement virtual cluster technologies (vCluster, Hierarchical Namespace Controller, or equivalent) to achieve API-server-level isolation between tenants while sharing underlying infrastructure, and must document the isolation guarantees and residual risks of the chosen approach.

## Objective

Provide stronger tenant isolation than namespace boundaries alone, enabling multiple teams or customers to share physical infrastructure with API-server-level separation.

## Rationale

Namespace isolation has inherent limitations: cluster-scoped resources (CRDs, ClusterRoles, PriorityClasses) are shared, and a tenant with sufficient RBAC permissions can affect cluster-wide behavior. Virtual clusters provide each tenant with an isolated control plane, limiting the impact of misconfigurations and privilege escalation attempts to a single virtual cluster. This is particularly important when different tenants operate at different trust levels or handle data of different classification levels.

## Scope

Applies to multi-tenant Kubernetes environments where namespace-level isolation is insufficient for the organization's threat model, such as environments with external tenants, cross-business-unit sharing, or mixed data classification levels. Not required for single-tenant or dedicated-cluster deployments. The chosen technology and its isolation guarantees must be documented.

## Enforcement

- Primary Enforcement Point: Virtual cluster provisioning system (vCluster Helm charts, HNC configuration), host cluster admission control restricting tenant access to the host API server, and periodic isolation validation tests.
- Implementation Surface: vCluster deployments providing virtual API server, scheduler, and CRD registry per tenant, Hierarchical Namespace Controller (HNC) for namespace hierarchy with delegated administration and policy inheritance, dedicated cluster provisioning for full physical separation, residual risk documentation for each isolation technology, and node-level controls (D2) as compensating controls for shared-kernel risks in virtual cluster environments.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Cloud Platform Engineering
