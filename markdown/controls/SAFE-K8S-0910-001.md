# SAFE-K8S-0910-001 - Multi-cluster AI workload orchestration security

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.10 - Multi-Cluster and Federated AI Security
- Maturity: Advanced
- Classification: ai-specific

## Statement

Multi-cluster AI workload orchestration (cross-cluster training coordination, multi-cluster inference routing, and multi-cluster pipeline scheduling) must enforce authenticated and authorized cross-cluster API calls, must verify cluster health and security posture before workload placement, and must implement circuit-breaker controls that isolate compromised clusters from the federation.

## Objective

Ensure that multi-cluster AI workload placement decisions are security-aware and that compromised clusters can be rapidly isolated from the federation without disrupting AI workloads on healthy clusters.

## Rationale

Multi-cluster AI orchestration introduces a control plane that spans cluster boundaries. Without authenticated cross-cluster API calls, an attacker who compromises one cluster's orchestration credentials can redirect workloads to attacker-controlled infrastructure. Without security-aware placement, workloads may be scheduled on clusters with known vulnerabilities. Without circuit-breaker isolation, a single compromised cluster can affect the entire federation through poisoned model updates, exfiltrated training data, or manipulated inference routing.

## Scope

Applies to organizations operating AI workloads across multiple Kubernetes clusters, whether on-premises, multi-cloud, or hybrid. Covers cross-cluster training coordination, multi-cluster inference routing, and multi-cluster pipeline scheduling. Does not cover single-cluster workload scheduling, which is addressed by SAFE-K8S-0901-006.

## Enforcement

- Primary Enforcement Point: Multi-cluster orchestration control plane configuration, cross-cluster API authentication settings, security-aware placement policy definitions, and circuit-breaker isolation trigger configuration.
- Implementation Surface: Multi-cluster orchestration control planes (Admiralty, Liqo, Kubernetes Federation v2, custom controllers), cross-cluster API authentication via SPIFFE/SPIRE federation or cloud-native workload identity federation (short-lived credentials), security-aware placement policies verifying target cluster Kubernetes version, admission controller status, GPU driver versions, and security posture assessments, circuit-breaker isolation triggers revoking federation credentials and draining workloads from compromised clusters, and centralized cross-cluster placement audit logs.
- Owner Primary: Platform Engineering
- Owner Supporting: ML Platform Engineering; Security Engineering; SRE
