# SAFE-K8S-0304-001 - Cloud Native 8 secure defaults enforcement

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.4 - Secure Defaults and Resource Constraints
- Maturity: Foundational
- Classification: baseline

## Statement

All Kubernetes workloads must comply with the CNCF Cloud Native 8 secure defaults: non-root execution, read-only root filesystem, minimal capabilities, no privilege escalation, resource limits defined, service account token opt-out where unused, seccomp profile applied, and no host namespace sharing.

## Objective

Establish a comprehensive baseline security posture for all Kubernetes workloads by enforcing the eight CNCF-recommended secure defaults as a non-negotiable minimum standard.

## Rationale

The Cloud Native 8 represents the consensus minimum security configuration endorsed by the cloud native community. Workloads that violate any of these defaults carry known, well-documented risks including container escape, privilege escalation, and lateral movement. Enforcing all eight defaults as a cohesive set prevents the common pattern of hardening some aspects while leaving others open, which creates a false sense of security.

## Scope

Applies to all workloads in production and staging clusters. Each of the eight defaults must be enforced unless a documented exception exists. Managed Kubernetes services must verify that provider-managed components do not violate these defaults. System-level workloads in kube-system may require documented exceptions for specific defaults.

## Enforcement

- Primary Enforcement Point: Pod Security Admission (Restricted level covers most defaults), policy engine rules covering remaining defaults not fully addressed by PSA, and CI/CD pipeline manifest validation.
- Implementation Surface: Pod Security Admission (Restricted level covering non-root, minimal capabilities, no privilege escalation, seccomp, no host namespaces), policy engine rules (OPA Gatekeeper, Kyverno) enforcing read-only root filesystem, resource limits, and SA token opt-out not covered by PSA, LimitRange objects for resource defaults, ServiceAccount automountServiceAccountToken field, CI/CD pipeline manifest validation for Cloud Native 8 compliance, and cluster-wide compliance scanning scripts (jq-based kubectl audits).
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
