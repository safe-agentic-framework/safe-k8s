# SAFE-K8S-0403-010 - Kubernetes Secrets external KMS integration and least-privilege key policy

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.3 - Secrets Management
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes Secrets encryption at rest must use an external KMS provider as the key management backend, and KMS key access policies must enforce least-privilege access to encryption and decryption operations.

## Objective

Ensure the cryptographic keys protecting Kubernetes Secrets are managed outside the cluster trust boundary and are governed by appropriately restricted key access policies.

## Rationale

Externalizing key management only improves security if the key service itself is governed correctly. This control isolates KMS backend integration and least-privilege key policy so teams can independently verify that Secrets encryption relies on an external KMS with restricted key access.

## Scope

Applies to all production Kubernetes clusters. Covers KMS provider selection and integration, customer-managed key configuration, and least-privilege key access policies. The underlying API-resource encryption mechanism remains governed by SAFE-K8S-0101-003.

## Enforcement

- Primary Enforcement Point: KMS provider key policies and CMK integration configuration.
- Implementation Surface: Kubernetes EncryptionConfiguration with KMS provider plugins, KMS v2 API, external KMS key access policies, and customer-managed key configurations for managed Kubernetes services.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Key Management/Cryptography; Cloud Platform Engineering
