# SAFE-K8S-1001-002 - Cluster-level logging architecture

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.1 - Logging and Audit
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must implement a cluster-level logging architecture that collects node-level and container logs from all AI workloads, aggregates them into a centralized logging platform, enforces log integrity through cryptographic verification or append-only storage, and applies retention policies aligned with organizational and regulatory requirements.

## Objective

Provide a reliable, tamper-evident, centralized log repository that supports security monitoring, incident investigation, and compliance auditing for AI workloads running on Kubernetes.

## Rationale

Distributed Kubernetes environments produce logs across many nodes and containers that are ephemeral by nature. Without centralized collection, integrity protection, and defined retention, critical evidence of security incidents, such as unauthorized GPU access or training data exfiltration, can be lost when pods are evicted or nodes are recycled, undermining incident response and audit capabilities.

## Scope

Applies to all Kubernetes clusters hosting AI workloads in production and pre-production environments. Includes node-level log agents, centralized log aggregation services, and log storage backends. Managed Kubernetes services must forward provider-generated logs to the organization's centralized platform.

## Enforcement

- Primary Enforcement Point: Logging agent DaemonSet configuration, centralized log aggregation platform settings, and log retention policy enforcement.
- Implementation Surface: DaemonSet-based log collectors (Fluent Bit, Fluentd, or Vector) deployed on all nodes including GPU nodes with appropriate taint tolerations, container log paths (/var/log/containers/), kubelet logs (journald or /var/log/), GPU driver logs (/var/log/nvidia-* or journald NVIDIA identifier), centralized log aggregation platforms (Elasticsearch, OpenSearch, Loki, Splunk), append-only or WORM-compliant object storage (S3 Object Lock, GCS retention policies, Azure Blob immutability policies), cryptographic log integrity verification (hash chains, signed log batches), and retention policy configuration per log class (audit logs, training logs, inference logs, GPU hardware logs, pipeline logs).
- Owner Primary: Security Operations
- Owner Supporting: Platform Engineering; Infrastructure Engineering
