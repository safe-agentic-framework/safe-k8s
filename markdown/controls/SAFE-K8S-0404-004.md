# SAFE-K8S-0404-004 - Certificate rotation, expiry monitoring, and compromise response

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.4 - Certificate Management
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must automate certificate rotation before expiry, implement monitoring and alerting for certificates approaching expiration thresholds, and maintain incident response procedures for revoking and re-issuing certificates when private keys are compromised.

## Objective

Prevent service outages from expired certificates and ensure rapid containment when certificate private keys are compromised, maintaining continuous transport-layer security across the cluster.

## Rationale

Expired certificates cause immediate service disruptions as TLS handshakes fail, often leading to emergency responses that bypass security controls (e.g., disabling TLS verification). Without automated rotation, organizations depend on manual tracking of potentially hundreds of certificates across a cluster. When a private key is compromised, the ability to rapidly revoke and re-issue the affected certificate limits the window during which an attacker can impersonate the affected service.

## Scope

Applies to all TLS certificates in the cluster including API server certificates, etcd peer certificates, kubelet certificates, webhook certificates, service mesh certificates, and application certificates. Covers automated rotation configuration, expiry monitoring thresholds, alerting integrations, and documented incident response procedures for key compromise scenarios.

## Enforcement

- Primary Enforcement Point: cert-manager renewal configurations with renewBefore thresholds, certificate expiry monitoring dashboards and alerts, and documented certificate compromise incident response procedures with revocation and re-issuance steps.
- Implementation Surface: cert-manager Certificate resources with renewBefore threshold configurations, certificate expiry monitoring dashboards and alerting (Prometheus cert-manager metrics, custom expiry scanners), CRL and OCSP revocation infrastructure, documented certificate compromise incident response procedures (revocation, re-issuance from new key pair, service redeployment verification), and monitoring for API server, etcd peer, kubelet, webhook, and service mesh certificate lifecycle.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Security Operations
