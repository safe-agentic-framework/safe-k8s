# SAFE-K8S-0606-006 - Kubernetes manifest signing and verification

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.6 - CI/CD and GitOps Pipeline Security
- Maturity: Advanced
- Classification: baseline

## Statement

Kubernetes deployment manifests must be cryptographically signed before deployment and verified at admission time. Admission controllers must reject manifests that lack a valid signature from an authorized signing identity.

## Objective

Extend supply chain integrity assurance to deployment manifests, preventing unauthorized modification of resource limits, security contexts, environment variables, or image references after image signing.

## Rationale

Signing container images but not the manifests that deploy them leaves a gap: an attacker can modify resource limits, security contexts, environment variables, or image references in manifests without detection. Manifest signing closes this gap by providing tamper-evident assurance that the deployment configuration was authored and approved through the standard review process.

## Scope

Applies to all Kubernetes manifests used for production and staging deployments. Development-only environments may have reduced signing requirements with documented justification.

## Enforcement

- Primary Enforcement Point: CI/CD pipeline manifest signing step and admission controller manifest signature verification.
- Implementation Surface: Kubernetes manifest cryptographic signing (cosign, GPG) and admission controller manifest signature verification, Git commit signing enforcement (GPG/SSH signatures) on deployment branches via Flux verify and ArgoCD GPG verification, and CI/CD pipeline manifest signing steps integrated with the build and promotion workflow.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: Platform Engineering; Security Engineering
