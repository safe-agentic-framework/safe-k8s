# SAFE-K8S-1007-002 - Kubernetes API deprecation and removal tracking

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.7 - Cluster Lifecycle and Asset Inventory
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must track Kubernetes API version deprecations and removals by monitoring official Kubernetes release channels and CHANGELOG entries, maintaining a registry of deprecated APIs used in the cluster, and ensuring teams receive advance notice of scheduled API removals before they affect production workloads.

## Objective

Prevent AI workload failures caused by undetected Kubernetes API removals by establishing systematic deprecation monitoring and advance notification before production upgrades.

## Rationale

Kubernetes removes deprecated APIs on a defined schedule. AI workloads that use custom resources and operators referencing specific API versions will fail if those APIs are removed in a cluster upgrade without prior detection. Systematic deprecation tracking ensures migration work is identified and planned before API removals cause production failures in AI operators, training job controllers, or inference services.

## Scope

Applies to all Kubernetes clusters scheduled for version upgrades. Workload scanning must cover all namespaces including AI workload namespaces, operator deployments, and Helm chart templates. Compatibility testing must validate AI-specific custom resources and operator functionality against the target Kubernetes version.

## Enforcement

- Primary Enforcement Point: Deprecation scanning tools in CI/CD pipeline, pre-upgrade compatibility test gates, and upgrade approval checklist.
- Implementation Surface: API deprecation scanning tools (kubent, pluto, kubepug, kubectl deprecations), AI operator manifest and Helm chart scanning for deprecated API versions (Kubeflow operators, KServe/KNative manifests, volcano scheduler configurations, NVIDIA GPU operator charts), CRD definition deprecated field detection, Kubernetes changelog removal list review, test cluster at target Kubernetes version (kind or minikube), CI-based compatibility testing deploying AI operators and workloads against target version, integration tests exercising training job creation, inference service deployment, GPU allocation, policy enforcement, and admission webhook validation against new API server versions.
- Owner Primary: Platform Engineering
- Owner Supporting: AI/ML Engineering; DevOps; Security Engineering
