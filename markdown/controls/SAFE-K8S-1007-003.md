# SAFE-K8S-1007-003 - AI infrastructure asset inventory and classification

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.7 - Cluster Lifecycle and Asset Inventory
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must maintain an automated, continuously updated inventory of all AI infrastructure assets running on Kubernetes, including models, datasets, ML pipelines, inference endpoints, and GPU allocations, with each asset classified by sensitivity, business criticality, and regulatory exposure.

## Objective

Provide comprehensive visibility into the AI asset landscape on Kubernetes to support security monitoring, incident response, access control decisions, and compliance reporting.

## Rationale

AI assets (trained models, datasets, inference endpoints) proliferate rapidly across Kubernetes namespaces and clusters. Without automated discovery and classification, organizations lose track of deployed models, orphaned GPU allocations, and unprotected inference endpoints, creating shadow AI infrastructure that is not covered by security monitoring, access controls, or compliance programs. Manual inventory methods cannot keep pace with the velocity of AI workload deployments.

## Scope

Applies to all Kubernetes clusters hosting AI workloads. Covers model artifacts in registries and running deployments, training datasets in persistent volumes and object storage, ML pipelines, inference service endpoints, and GPU device allocations. Inventory must be reconciled at least daily through automated discovery.

## Enforcement

- Primary Enforcement Point: Asset inventory automation tooling, Kubernetes resource discovery agents, and classification policy enforcement.
- Implementation Surface: Automated asset discovery controller or CronJob scanning for AI-specific CRDs (InferenceService, TFJob, PyTorchJob, MPIJob, Argo Workflow, Kubeflow Pipeline), model registry API integration, PVC label scanning for dataset discovery, node labels and pod resource requests for GPU allocation tracking, external metadata enrichment (model registry version/accuracy, CI/CD pipeline run history, cloud provider GPU instance type/cost), queryable asset inventory storage (database, CMDB, or structured file with change tracking), classification label enforcement via admission policy (sensitivity, criticality, regulatory-exposure labels), and unclassified asset alerting.
- Owner Primary: Platform Engineering
- Owner Supporting: AI/ML Engineering; Security Operations; GRC/Compliance
