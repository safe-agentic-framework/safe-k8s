# SAFE-K8S-1005-002 - Kubernetes and AI workload containment strategies

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.5 - Incident Response for Kubernetes
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must define and document containment strategies for Kubernetes security incidents, including node draining, namespace isolation via network policies, workload suspension, and credential revocation, and must extend these strategies with AI-specific containment actions including GPU node draining, inference service quarantine, and pipeline execution suspension.

## Objective

Enable rapid, pre-planned containment of security incidents in Kubernetes AI environments that limits blast radius while preserving forensic evidence and minimizing disruption to unaffected workloads.

## Rationale

Kubernetes environments require containment actions that differ fundamentally from traditional infrastructure, such as network policy-based namespace isolation rather than firewall rules, or pod eviction rather than server shutdown. AI workloads add further complexity: draining a GPU node affects all co-scheduled training jobs, and suspending a pipeline mid-execution can corrupt training state. Without pre-defined containment procedures, responders may apply overly broad actions that cause unnecessary outages or insufficiently targeted actions that fail to contain the threat.

## Scope

Applies to all Kubernetes clusters hosting AI workloads. Containment procedures must address node-level, namespace-level, and workload-level isolation. AI-specific containment must cover GPU node operations, inference services, and ML pipeline controllers. Procedures must account for multi-tenant environments where containment actions may affect co-located workloads.

## Enforcement

- Primary Enforcement Point: IR playbook containment procedures, network policy templates for isolation, and RBAC configuration for emergency containment actions.
- Implementation Surface: kubectl cordon and kubectl drain commands with GPU-aware options (--grace-period for checkpoint save, --pod-selector for selective draining, --ignore-daemonsets), deny-all NetworkPolicy manifests for namespace-level isolation (with logging egress exceptions), Calico HostEndpoint policies for node-level network isolation, InferenceService replica scaling (scale to zero for quarantine), Argo Workflow suspension and CronJob suspend:true settings, training job CRD deletion/suspension with checkpoint preservation, ServiceAccount token deletion and credential rotation procedures, and documented containment strategy runbooks for each isolation layer.
- Owner Primary: Security Operations
- Owner Supporting: Platform Engineering; AI/ML Engineering; SRE
