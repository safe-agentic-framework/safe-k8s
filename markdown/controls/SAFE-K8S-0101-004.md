# SAFE-K8S-0101-004 - API server access controls and authorization modes

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.1 - Kubernetes API Server Security
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must enable RBAC and Node authorization, must not use ABAC or AlwaysAllow, and must allow Webhook authorization only for approved, authenticated, and integrity-protected endpoints.

## Objective

Ensure API server access decisions are governed by a structured, auditable authorization model that enforces least privilege and prevents legacy or misconfigured authorization from granting unintended access.

## Rationale

Kubernetes evaluates authorization modes in order (--authorization-mode flag). The standard production pattern is Node,RBAC, which ensures kubelets are scoped to their own node objects while all other access is governed by auditable RBAC bindings. ABAC policies are difficult to audit and easy to misconfigure, creating authorization gaps that RBAC eliminates. Unauthenticated or unprotected webhook authorizer endpoints introduce external authorization dependencies without integrity guarantees.

## Scope

Applies to all production clusters. Covers the authorization mode chain (--authorization-mode flag), ABAC policy file removal, AlwaysAllow prohibition, Node and RBAC authorizer configuration, and webhook authorizer endpoint security including TLS and authentication. For managed Kubernetes services, applies to customer-configurable authorization settings.

## Enforcement

- Primary Enforcement Point: RBAC / Authorization
- Implementation Surface: API server authorization mode configuration, RBAC role and binding definitions, and periodic authorization configuration assessment.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Identity and Access Management
