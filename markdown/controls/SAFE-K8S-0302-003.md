# SAFE-K8S-0302-003 - Privilege escalation prevention controls

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.2 - Security Contexts and Capabilities
- Maturity: Foundational
- Classification: baseline

## Statement

All containers must set the no-new-privileges flag (allowPrivilegeEscalation: false), must run with explicitly assigned non-root UID/GID pairs, and must not have filesystem ownership configurations (fsGroup, supplementalGroups) that grant write access to sensitive paths outside the container's designated data directories.

## Objective

Prevent processes within containers from acquiring additional privileges beyond those granted at container start, closing privilege escalation paths through setuid binaries, kernel exploits, and filesystem ownership manipulation.

## Rationale

The no-new-privileges flag prevents child processes from gaining more privileges than their parent through setuid/setgid binaries or capability-raising execve calls. Without explicit UID/GID assignment, containers may run as root or use UIDs that collide with host users. Misconfigured fsGroup or supplementalGroups can grant container processes write access to volumes shared with other workloads, enabling cross-workload data tampering.

## Scope

Applies to all pods and containers in production clusters, including init containers and ephemeral debug containers. UID/GID assignments must not conflict with host system users or with other containers sharing the same volumes.

## Enforcement

- Primary Enforcement Point: Pod Security Admission (Restricted level), policy engine rules validating securityContext.allowPrivilegeEscalation, runAsUser, runAsGroup, and fsGroup settings.
- Implementation Surface: Pod securityContext fields (allowPrivilegeEscalation, runAsUser, runAsGroup, fsGroup, supplementalGroups), Pod Security Admission (Restricted level), policy engine rules validating no-new-privileges flag and UID/GID assignments, volume permission analysis tooling, and container image UID verification.
- Owner Primary: Application Development teams
- Owner Supporting: Platform Engineering; Security Engineering
