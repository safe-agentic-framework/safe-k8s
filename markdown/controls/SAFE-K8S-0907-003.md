# SAFE-K8S-0907-003 - Training dataset and model deployment access controls

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.7 - Feature Store Security and Data Access Controls
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Access to training datasets stored in persistent volumes, object storage, and feature stores must be restricted through Kubernetes-native and external authorization controls, and model download and deployment operations must be authorized to control which principals can pull specific models and deploy to specific namespaces.

## Objective

Enforce least-privilege access to training data and model artifacts by controlling both data access for training and model deployment authorization across Kubernetes namespaces.

## Rationale

Training datasets contain sensitive data and proprietary information; without fine-grained access controls, any workload in the cluster can read training data. Without model deployment authorization, any user with namespace access can deploy any model, enabling deployment of untested or malicious models. The combination of uncontrolled data access and unrestricted model deployment creates a path from data theft through model tampering to production compromise.

## Scope

Applies to all training data storage (persistent volumes, object storage, feature stores) and model deployment workflows on Kubernetes. Covers Kubernetes RBAC, storage-level access policies, external authorization (OPA, Kyverno), and namespace-scoped model deployment controls. Excludes model registry access controls covered by SAFE-K8S-0905-003.

## Enforcement

- Primary Enforcement Point: Kubernetes RBAC and storage access policies for training data, and admission control or deployment pipeline policies governing model pull and namespace deployment authorization.
- Implementation Surface: Kubernetes RBAC restricting PVC mounts to authorized training namespace service accounts, cloud IAM policies via workload identity federation for object storage access (S3, GCS), feature store RBAC for per-feature-view access control, model registry pull access controls per service account, admission control (OPA Gatekeeper, Kyverno) verifying model promotion gate passage before deployment, and namespace-scoped deployment policies restricting model deployment to authorized namespaces.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Data Engineering; Security Engineering; Platform Engineering
