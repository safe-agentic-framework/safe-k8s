# SAFE-K8S-0503-002 - DNS infrastructure hardening

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.3 - Ingress, Egress, and DNS Hardening
- Maturity: Practitioner
- Classification: baseline

## Statement

CoreDNS must be configured with hardened settings including restricted upstream resolver access, DNS policy enforcement must prevent unauthorized DNS resolution paths, and DNS-based service discovery must be limited to authorized namespaces.

## Objective

Secure the cluster DNS infrastructure to prevent DNS-based attacks and unauthorized service discovery by hardening CoreDNS configuration and enforcing DNS resolution policies.

## Rationale

DNS is a required service in every Kubernetes cluster and a high-value attack target. Misconfigured CoreDNS can expose internal service names to unauthorized parties. Without DNS policy enforcement, workloads can resolve arbitrary external domains, bypassing egress controls.

## Scope

Applies to CoreDNS deployments in all production clusters. Includes upstream DNS resolver configuration, DNS policy enforcement (via CNI or external DNS proxy), and namespace-scoped service discovery restrictions. Node-local DNS caches (NodeLocal DNSCache) must also adhere to hardening requirements.

## Enforcement

- Primary Enforcement Point: CoreDNS Corefile configuration, DNS network policies, and CNI DNS-aware policy enforcement.
- Implementation Surface: CoreDNS Corefile configuration (log plugin, forward directive to trusted upstream resolvers, cache settings, kubernetes plugin), DNS egress NetworkPolicy restricting pods to CoreDNS only, Cilium DNS proxy and DNS-aware policies, and NodeLocal DNSCache hardening.
- Owner Primary: Network Engineering
- Owner Supporting: Platform Engineering
