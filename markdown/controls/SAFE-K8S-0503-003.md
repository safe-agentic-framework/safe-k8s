# SAFE-K8S-0503-003 - External traffic policy and source IP preservation

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.3 - Ingress, Egress, and DNS Hardening
- Maturity: Practitioner
- Classification: baseline

## Statement

Services exposed to external traffic must use externalTrafficPolicy settings appropriate for the security requirements, source IP addresses must be preserved where client IP-based access controls or audit logging require them, and traffic routing decisions between Local and Cluster mode must be documented with security tradeoffs.

## Objective

Ensure that external traffic routing preserves client IP visibility needed for access control and forensic analysis while maintaining acceptable availability characteristics.

## Rationale

The default Cluster externalTrafficPolicy performs SNAT, replacing the client source IP with a node IP, which destroys client identity information needed for IP-based access controls, geographic restrictions, and audit trails. Without source IP preservation, security teams cannot accurately attribute requests to origin, enforce IP allowlists, or investigate incidents involving external traffic. Choosing Local policy without understanding its availability tradeoffs can cause traffic blackholing when pods are unevenly distributed.

## Scope

Applies to all Services of type LoadBalancer or NodePort in production clusters that receive external traffic. The choice between Local and Cluster externalTrafficPolicy must be documented per service with security and availability justification.

## Enforcement

- Primary Enforcement Point: Kubernetes Service spec externalTrafficPolicy field, cloud load balancer configuration for proxy protocol or X-Forwarded-For header preservation.
- Implementation Surface: Kubernetes Service spec externalTrafficPolicy field (Local vs. Cluster mode), cloud load balancer proxy protocol configuration, X-Forwarded-For header preservation settings, pod topology and scheduling constraints for Local traffic policy availability, and documentation of per-service traffic policy decisions with security and availability tradeoffs.
- Owner Primary: Network Engineering
- Owner Supporting: Application Development; Security Engineering
