# SAFE-K8S-0504-003 - SPIFFE/SPIRE workload identity management

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.4 - Zero Trust Architecture and Service Mesh
- Maturity: Advanced
- Classification: baseline

## Statement

Workload identity must be established using SPIFFE-compliant identities issued by SPIRE or an equivalent workload identity provider, trust domains must be properly configured and scoped, and cross-cluster identity federation must use explicit trust bundle exchange rather than implicit trust.

## Objective

Provide cryptographically verifiable workload identities that enable identity-based access control independent of network location, with secure cross-cluster federation.

## Rationale

Without cryptographic workload identity, services rely on network-based trust (source IP, namespace labels) which can be spoofed or circumvented. SPIFFE provides a standardized, attestation-based identity framework that binds identity to workload properties rather than network location. Improperly configured trust domains or implicit cross-cluster trust allow workloads in one environment to impersonate identities from another, enabling unauthorized cross-cluster access.

## Scope

Applies to all production Kubernetes clusters implementing zero trust networking. SPIRE server and agent deployments must follow high-availability patterns. Trust domain naming must align with organizational boundaries. Cross-cluster federation applies where workloads in separate clusters need to authenticate to each other.

## Enforcement

- Primary Enforcement Point: SPIRE server and agent configuration, workload registration entries, trust bundle management, and federation endpoint configuration.
- Implementation Surface: SPIRE server and agent deployments (high-availability configuration), SPIFFE ID registration entries (spire-server entry create with k8s namespace and service account selectors), trust domain configuration and naming, SVID (SPIFFE Verifiable Identity Document) certificate issuance and automatic rotation, cross-cluster trust bundle exchange via SPIRE federation endpoints, workload attestation plugins (k8s_sat, k8s_psat node attestation), and integration with service mesh or application-level mTLS for identity-based access control.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Network Engineering
