# SAFE-K8S-0606-014 - GitOps release metadata preservation and deployment traceability

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.6 - CI/CD and GitOps Pipeline Security
- Maturity: Practitioner
- Classification: baseline

## Statement

GitOps deployment processes must preserve release metadata for deployed packages, versions, and configuration files so deployed state can be reconstructed and traced during audits, incident response, and rollback analysis.

## Objective

Ensure GitOps-managed deployments retain sufficient package, version, and configuration metadata to reconstruct what was deployed, when it changed, and which artifacts and revisions were involved.

## Rationale

Even when repository protections are strong, teams cannot investigate incidents, prove deployment history, or execute reliable rollback analysis if package versions, rendered configuration revisions, and release metadata are not preserved. Traceability evidence is a distinct audit and response obligation from repository hardening.

## Scope

Applies to production and staging GitOps deployments and to the deployment records, release manifests, package metadata, and configuration revision history needed to reconstruct deployed state.

## Enforcement

- Primary Enforcement Point: GitOps deployment history, release records, and configuration version metadata.
- Implementation Surface: Release manifests, package version records, Git revision metadata, deployment history retained by GitOps controllers, and artifact or environment metadata linking deployed packages and configuration files to specific revisions.
- Owner Primary: Platform Engineering
- Owner Supporting: DevSecOps / Release Engineering; Security Engineering
