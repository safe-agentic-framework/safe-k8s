# SAFE-K8S-0102-005 - etcd certificate rotation and lifecycle management

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters must implement automated rotation of etcd server certificates, peer certificates, and client certificates (used by the API server to connect to etcd) before expiration. Certificate validity periods must not exceed organizationally defined maximums. The rotation process must be tested and documented, and certificate expiration monitoring must generate alerts with sufficient lead time to prevent service disruption.

## Objective

Prevent cluster outages and security degradation caused by expired or long-lived etcd certificates by ensuring timely, automated rotation with expiration monitoring.

## Rationale

etcd certificates secure all communication between etcd peers and between the API server and etcd. Expired certificates cause immediate cluster failure as the API server loses connectivity to its backing store. Long- lived certificates increase the window of exposure if a private key is compromised. Manual rotation processes are error-prone and often neglected, particularly in long-running clusters. Automated rotation with monitoring ensures continuity and reduces the risk of certificate-related outages.

## Scope

Applies to all self-managed Kubernetes clusters using etcd as the backing store. Covers etcd server TLS certificates, peer TLS certificates, and API server-to-etcd client certificates. For managed Kubernetes services, this control is provider-managed; verify provider documentation for certificate rotation practices and SLAs.

## Enforcement

- Primary Enforcement Point: Certificate Management / PKI
- Implementation Surface: Certificate management tooling (kubeadm certs, cert-manager, custom PKI), etcd static pod manifest certificate paths, monitoring and alerting for certificate expiration (Prometheus cert-exporter or equivalent).
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
