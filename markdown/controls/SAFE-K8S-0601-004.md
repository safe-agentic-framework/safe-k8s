# SAFE-K8S-0601-004 - Image hardening practices

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.1 - Container Image and Registry Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Container images must run as non-root users, mount filesystems as read-only where operationally feasible, exclude shell binaries and unnecessary utilities, and minimize the overall attack surface. Inference-serving images must use distroless or multi-stage build outputs that retain only the runtime libraries required for GPU compatibility while eliminating compilers, package managers, and debugging tools.

## Objective

Harden container images at build time to eliminate unnecessary capabilities and reduce the blast radius of a container compromise, including inference-specific hardening that preserves GPU functionality.

## Rationale

Images running as root or containing shell binaries provide adversaries with immediate post-exploitation capabilities such as privilege escalation, filesystem modification, and lateral movement tooling. AI inference images are particularly high-value targets; without hardening, a compromised inference container could expose model weights, training data references, or GPU memory contents. Removing unnecessary tooling limits what an attacker can accomplish after initial access.

## Scope

Applies to all container images deployed in production and pre-production Kubernetes clusters. AI inference and training images must balance hardening with GPU driver and CUDA runtime compatibility requirements. Documented exceptions are permitted for debugging images in non-production namespaces with time-limited access.

## Enforcement

- Primary Enforcement Point: CI/CD pipeline image build validation and admission controller policies enforcing non-root, read-only filesystem, and capability drop constraints.
- Implementation Surface: Dockerfile USER directive (non-root UID), readOnlyRootFilesystem pod securityContext settings, Kubernetes emptyDir volume mounts for writable paths, admission controller policies enforcing non-root and read-only constraints (Kyverno, OPA/Gatekeeper), multi-stage Dockerfile build outputs with shell binary and package manager removal, ldd verification of final image library chains, NVIDIA runtime .so file copying for GPU compatibility, and CI/CD pipeline image build validation steps.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: ML Platform Engineering; Platform Engineering
