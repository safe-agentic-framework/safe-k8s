# SAFE-K8S-0402-001 - Default service account restriction

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.2 - Service Accounts and Workload Identity
- Maturity: Foundational
- Classification: baseline

## Statement

Workloads must not use the default service account in any namespace. Auto-mounting of service account tokens must be disabled on the default service account, and each workload must be assigned a dedicated service account with only the permissions required for its function.

## Objective

Eliminate the use of shared default service accounts that provide a common attack surface across all pods in a namespace, ensuring each workload operates with a distinct identity and minimal permissions.

## Rationale

The default service account in each namespace is automatically assigned to all pods that do not specify one. If any pod in the namespace is compromised, the attacker inherits whatever permissions have been granted to the default service account. Disabling auto-mount and assigning dedicated per-workload service accounts limits lateral movement and ensures that token compromise affects only the specific workload's permissions.

## Scope

Applies to all namespaces in all production and pre-production clusters. System namespaces (kube-system, kube-public) may retain default service account usage for system components with documented justification. Managed Kubernetes services must configure this for all customer namespaces.

## Enforcement

- Primary Enforcement Point: Pod admission policies that reject pods using the default service account or mounting tokens without explicit opt-in, combined with namespace-level service account configuration.
- Implementation Surface: ServiceAccount resource configuration (automountServiceAccountToken: false on default service account), Pod spec serviceAccountName field, admission policies (OPA Gatekeeper, Kyverno, Pod Security Admission) rejecting pods using the default service account or omitting serviceAccountName, and namespace-level service account provisioning automation.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development
