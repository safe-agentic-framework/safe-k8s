# SAFE-K8S-0203-005 - Kernel module restriction and lockdown mode

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.3 - Host OS and Kernel Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes nodes must restrict kernel module loading by disabling unnecessary modules, blacklisting known-risky modules via /etc/modprobe.d/ configuration, and enabling kernel lockdown mode where supported to prevent runtime loading of unsigned or unauthorized kernel modules.

## Objective

Prevent unauthorized kernel module loading that could bypass container isolation, install kernel-level rootkits, or introduce unvetted code into the kernel execution environment.

## Rationale

Unrestricted kernel module loading allows post-exploitation installation of rootkits, keyloggers, or network sniffers that operate below container isolation boundaries. Kernel lockdown mode prevents even root-level processes from modifying kernel memory or loading unsigned modules, providing defense-in-depth against host compromise. Nodes running GPU workloads require specific kernel modules (e.g., NVIDIA drivers) that must be explicitly whitelisted and signature-verified.

## Scope

Applies to all production cluster nodes regardless of boot verification support. Nodes requiring specific kernel modules (e.g., GPU drivers, InfiniBand) must whitelist only required modules and document exceptions. DKMS-signed modules must have their signing keys validated.

## Enforcement

- Primary Enforcement Point: Kernel module blacklist configuration in /etc/modprobe.d/ and kernel lockdown mode settings applied during node image build or provisioning.
- Implementation Surface: Kernel module blacklisting (/etc/modprobe.d/), NVIDIA kernel module signing verification, kernel lockdown mode settings, DKMS-signed module management, approved module whitelist, and node image build pipeline module verification steps.
- Owner Primary: OS/Infrastructure
- Owner Supporting: Platform Engineering; Security Engineering
