# SAFE-K8S-0703-011 - EDoS prevention and chaos engineering for AI resources

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.3 - Resource Governance and Priority
- Maturity: Advanced
- Classification: ai-specific

## Statement

Organizations must implement Economic Denial of Sustainability (EDoS) controls for cloud-hosted AI infrastructure including cloud auto-scaling maximum limits and spending guardrails, and must conduct periodic chaos engineering tests against GPU resource controls, queue management, and auto-scaling limits to validate resilience.

## Objective

Prevent uncontrolled cloud spending from AI workload auto-scaling and validate resource control effectiveness through structured failure injection testing.

## Rationale

Without EDoS controls, an attacker or misconfigured workload can trigger auto-scaling that consumes cloud budgets. Without resilience testing (chaos engineering), organizations discover resource control failure modes in production rather than in controlled test environments. Chaos engineering validates that bounded queues, PriorityClass configurations, and PDB settings function correctly under GPU device failure, OOM, and scheduler saturation scenarios.

## Scope

Applies to cloud-hosted Kubernetes clusters with auto-scaling enabled for AI workloads. Covers cloud spending guardrails, auto-scaling limits, and periodic chaos engineering test schedules targeting GPU resource controls and queue management.

## Enforcement

- Primary Enforcement Point: Cloud auto-scaling maximum limits, spending guardrails with automatic enforcement, and chaos engineering test schedules.
- Implementation Surface: Cloud provider spending alerts with automatic enforcement (scale-down triggers, job suspension, namespace quota reduction), chaos engineering tooling targeting GPU device failure, training job OOM, scheduler saturation, and queue backlog scenarios, PDB and PriorityClass validation under failure conditions, and documented recovery procedures for each failure mode.
- Owner Primary: SRE
- Owner Supporting: Platform Engineering; Security Engineering; FinOps
