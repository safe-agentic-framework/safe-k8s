# SAFE-K8S-0205-001 - kube-proxy mode selection and security hardening

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.5 - kube-proxy and Node Networking Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters must select and document the kube-proxy mode (iptables, IPVS, or eBPF-based replacement) based on a security and performance tradeoff analysis, and must harden the selected mode by restricting kube-proxy permissions, enabling strict conntrack settings, and removing unnecessary service proxy capabilities.

## Objective

Ensure that the service proxy layer is consciously selected and hardened based on the cluster's security requirements, rather than relying on default kube-proxy settings that may expose unnecessary attack surface.

## Rationale

kube-proxy manages service-level network routing on each node through iptables rules, IPVS virtual servers, or eBPF programs. Each mode has distinct security characteristics: iptables rules are difficult to audit at scale, IPVS exposes additional kernel surface, and eBPF replacements shift security policy to kernel programs. An unconsidered default selection may expose unnecessary kernel attack surface, create unauditable networking rules, or conflict with network policy enforcement. Deliberate mode selection with documented tradeoffs ensures the networking layer aligns with the cluster's security posture.

## Scope

Applies to all production clusters using kube-proxy or kube-proxy replacement implementations. Clusters using eBPF-based CNI plugins that fully replace kube-proxy (e.g., Cilium kube-proxy replacement) must document the replacement and apply equivalent hardening. Managed Kubernetes services must document the provider-selected mode and any available hardening options.

## Enforcement

- Primary Enforcement Point: Cluster provisioning templates and kube-proxy ConfigMap or CNI configuration during cluster bootstrap.
- Implementation Surface: kube-proxy mode configuration (iptables, IPVS, eBPF-based replacement), kube-proxy RBAC (system:node-proxier ClusterRole), conntrack tuning settings, eBPF replacement DaemonSet management (Cilium), and architecture decision records.
- Owner Primary: Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering
