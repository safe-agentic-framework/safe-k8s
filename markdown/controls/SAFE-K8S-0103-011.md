# SAFE-K8S-0103-011 - Control-plane high availability topology and failover

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.3 - Controller-Manager, Scheduler, and Cloud Controller Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters must deploy control-plane components in a highly available configuration with a minimum of three replicas distributed across distinct failure domains, configure etcd with an odd number of members sufficient for quorum, and expose the API server through a load balancer with automated health-check failover to eliminate single points of failure.

## Objective

Maintain control-plane availability during node, zone, or component failures by enforcing resilient control-plane topology and failover paths.

## Rationale

Leader election does not preserve availability if the control plane itself is deployed as a single point of failure. This control isolates the topology, quorum, and failover architecture obligation so operators can independently verify replica placement, etcd quorum design, and API endpoint failover behavior.

## Scope

Applies to production clusters. Covers control-plane replica count and placement, etcd quorum topology, and API server failover/load-balancing architecture. For managed Kubernetes, verify provider HA guarantees and exposed failover architecture.

## Enforcement

- Primary Enforcement Point: High Availability Architecture
- Implementation Surface: Control-plane topology inventory, replica placement across failure domains, etcd member configuration, load balancer health checks, and availability SLO dashboards.
- Owner Primary: Platform Engineering
- Owner Supporting: Site Reliability Engineering; Security Engineering
