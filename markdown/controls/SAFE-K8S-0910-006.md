# SAFE-K8S-0910-006 - Cross-cluster orchestration identity and audit security

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.10 - Multi-Cluster and Federated AI Security
- Maturity: Advanced
- Classification: ai-specific

## Statement

Multi-cluster AI workload orchestration control planes (cross-cluster training coordination, multi-cluster inference routing, and multi-cluster pipeline scheduling) must enforce authenticated and authorized cross-cluster API calls using short-lived federated workload identities, and must centralize orchestration audit logs for cross-cluster placement and routing decisions.

## Objective

Ensure that only trusted orchestration components can issue cross-cluster workload actions and that those actions remain traceable after a member cluster compromise.

## Rationale

Multi-cluster AI orchestration introduces a control plane that spans cluster boundaries. If cross-cluster API calls rely on replicated kubeconfigs, long-lived service account tokens, or weak authorization, an attacker who compromises one cluster can redirect workloads or issue unauthorized federation actions from attacker-controlled infrastructure. Centralized audit logging is required because logs stored only on the orchestrating cluster may be destroyed or altered during cluster compromise.

## Scope

Applies to organizations operating AI workloads across multiple Kubernetes clusters, whether on-premises, multi-cloud, or hybrid. Covers cross-cluster training coordination, multi-cluster inference routing, and multi-cluster pipeline scheduling. Does not cover target-cluster health evaluation or federation isolation triggers, which are addressed by SAFE-K8S-0910-007.

## Enforcement

- Primary Enforcement Point: Multi-cluster orchestration control plane identity federation configuration, authorization policy definitions for cross-cluster APIs, and centralized audit logging configuration.
- Implementation Surface: Multi-cluster orchestration control planes (Admiralty, Liqo, Kubernetes Federation v2, custom controllers), cross-cluster API authentication via SPIFFE/SPIRE federation or cloud-native workload identity federation with short-lived credentials, authorization policies restricting which orchestration principals may place or route workloads to which clusters, centralized audit logging for cross-cluster placement and routing actions, and certificate rotation and revocation workflows for federated orchestration identities.
- Owner Primary: Platform Engineering
- Owner Supporting: ML Platform Engineering; Security Engineering; SRE
