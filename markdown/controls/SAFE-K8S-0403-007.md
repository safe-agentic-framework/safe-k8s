# SAFE-K8S-0403-007 - AI platform key hierarchy and KMS architecture

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.3 - Secrets Management
- Maturity: Advanced
- Classification: ai-specific

## Statement

Organizations operating AI platforms must implement a structured key hierarchy using envelope encryption with distinct key domains for training data encryption, model artifact encryption, model signing, and inference payload protection, with HSM-backed root keys, automated key rotation per domain, and key access policies enforcing separation between training, serving, and governance environments.

## Objective

Ensure that cryptographic key management for AI platforms follows defense-in-depth principles with key separation by function, so that compromise of one key domain (e.g., training data encryption) does not grant access to other domains (e.g., model signing or inference payloads).

## Rationale

AI platforms require multiple categories of cryptographic keys -- data encryption keys for training datasets, model encryption keys for artifacts at rest, signing keys for model provenance attestation, and transport keys for inference payloads. Using a single KMS key for all purposes conflates trust domains: a training pipeline that can decrypt data should not automatically be able to sign models for production deployment. SAFE-K8S-0403-001 establishes external KMS integration for Kubernetes Secrets; this control extends key management to the full AI platform key hierarchy with function-specific key domains and access policies.

## Scope

Applies to all AI platform cryptographic operations including training data encryption at rest, model artifact encryption in registries, model signing for provenance and integrity, checkpoint encryption, and inference request/response payload protection. Covers KMS key hierarchy design, key access policy separation, HSM integration for root keys, and key rotation scheduling per domain.

## Enforcement

- Primary Enforcement Point: KMS key policies restricting each key's usage to specific IAM roles and workload identities, with separate keys for data encryption, model encryption, model signing, and inference protection.
- Implementation Surface: Cloud KMS (AWS KMS multi-key, GCP KMS key rings, Azure Key Vault), HSM-backed root keys, model registry encryption configuration, training pipeline data encryption integration, model signing infrastructure (Sigstore/cosign), inference gateway TLS/payload encryption.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; ML Engineering; Key Management/Cryptography
