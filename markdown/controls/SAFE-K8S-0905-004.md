# SAFE-K8S-0905-004 - ML dependency vulnerability management and ML-BOM

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.5 - AI Supply Chain and Model Lifecycle
- Maturity: Advanced
- Classification: ai-specific

## Statement

Organizations must perform software composition analysis for ML frameworks (PyTorch, TensorFlow, JAX), CUDA toolkits, and Python ecosystem dependencies, and must generate ML Bills of Materials (ML-BOMs) capturing model provenance metadata including training data lineage, hyperparameters, framework versions, and training infrastructure identity as defined by CTA-2114.

## Objective

Identify and remediate known vulnerabilities in ML software dependencies and maintain comprehensive provenance records that enable impact analysis when vulnerabilities or data issues are discovered.

## Rationale

ML workloads depend on complex software stacks including ML frameworks, CUDA toolkits, and hundreds of Python packages. Without SCA, known CVEs in these dependencies remain unpatched and exploitable. Without ML-BOMs, organizations cannot determine which models are affected when a vulnerability is discovered in a framework version or when training data is found to be compromised, making incident response slow and incomplete.

## Scope

Applies to all ML software dependencies used in training and inference environments on Kubernetes. Covers ML frameworks, CUDA toolkits, Python packages, and their transitive dependencies. ML-BOM generation covers model provenance metadata per CTA-2114 requirements. Excludes application-layer dependencies not specific to ML workloads.

## Enforcement

- Primary Enforcement Point: CI/CD pipeline SCA scanning stages, container image build processes, and ML-BOM generation tooling integrated into model training and registration workflows.
- Implementation Surface: CI/CD pipeline SCA scanning (Trivy, Grype, Snyk) for ML frameworks (PyTorch, TensorFlow, JAX), CUDA toolkits, cuDNN, NCCL, and Python dependencies, ML-BOM generation tooling conforming to CTA-2114 capturing model identity, training data lineage (DVC), hyperparameters (MLflow, W&B), framework versions, training infrastructure metadata, and evaluation results, ML-BOM signing alongside model artifacts in OCI registries, and deserialization vulnerability tracking (torch.load pickle, SafeTensors migration).
- Owner Primary: Security Engineering
- Owner Supporting: ML Platform Engineering; DevOps Engineering
