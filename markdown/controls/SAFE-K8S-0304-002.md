# SAFE-K8S-0304-002 - CPU, memory, and ephemeral storage resource limits

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.4 - Secure Defaults and Resource Constraints
- Maturity: Foundational
- Classification: baseline

## Statement

All pods must define resource requests and limits for CPU, memory, and ephemeral storage, and namespaces must enforce LimitRange and ResourceQuota objects that prevent workloads from consuming unbounded resources or starving co-located workloads.

## Objective

Prevent resource exhaustion, denial-of-service conditions, and noisy- neighbor effects by ensuring every workload declares its resource requirements and is bounded by enforced limits at both the pod and namespace level.

## Rationale

Workloads without resource limits can consume all available CPU, memory, or disk on a node, causing eviction of other workloads and potential node instability. Attackers who compromise a container without resource limits can launch resource-exhaustion attacks (cryptomining, fork bombs, disk fill) that affect all workloads on the same node. Without LimitRange enforcement, developers can deploy pods with no limits set, bypassing namespace-level resource governance.

## Scope

Applies to all pods in production clusters. LimitRange objects must be configured in every namespace to set default limits for pods that do not declare them. ResourceQuota objects must be configured to cap aggregate namespace resource consumption. GPU resource requests (nvidia.com/gpu) should be included where applicable.

## Enforcement

- Primary Enforcement Point: LimitRange and ResourceQuota admission controllers (built-in), policy engine rules validating that all pods specify resource requests and limits, and CI/CD pipeline manifest validation.
- Implementation Surface: Pod spec resources.requests and resources.limits fields (cpu, memory, ephemeral-storage, nvidia.com/gpu), LimitRange objects (per-namespace default and max limits), ResourceQuota objects (per-namespace aggregate caps including GPU extended resources), policy engine rules rejecting pods without resource specifications, CI/CD pipeline manifest validation, resource utilization monitoring dashboards, and GPU-aware ResourceQuota for nvidia.com/gpu allocation per team namespace.
- Owner Primary: Platform Engineering
- Owner Supporting: Application Development teams; AI/ML Platform team (for GPU resource quotas)
