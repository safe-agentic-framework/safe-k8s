# SAFE-K8S-0905-008 - Public model quarantine and vetting workflow

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.5 - AI Supply Chain and Model Lifecycle
- Maturity: Advanced
- Classification: ai-specific

## Statement

Organizations must implement a quarantine and vetting workflow for models downloaded from public hubs (Hugging Face Hub, model zoos, community repositories), must scan third-party models for malicious payloads (embedded code, pickle exploits, steganographic payloads, backdoor triggers) before internal registry admission, must verify available provenance metadata (model cards, commit signatures, community trust scores), and must internally re-sign and re-register externally sourced models before deployment to maintain chain of custody.

## Objective

Prevent introduction of malicious, backdoored, or unvetted models from public sources into organizational infrastructure by enforcing quarantine scanning, provenance verification, and internal re-registration.

## Rationale

Public model hubs host community-contributed models with minimal security vetting. Attackers publish models containing pickle-based code execution payloads, steganographic payloads hidden in weight tensors, backdoor triggers that activate on specific inputs, or subtly poisoned weights. Without a quarantine workflow, engineers can directly pull unvetted models into production infrastructure.

## Scope

Applies to all models sourced from external public hubs and community repositories. Covers the quarantine pipeline (format scanning, static analysis, sandbox execution, behavioral validation), provenance verification (publisher identity, commit signing, community trust signals), and internal re-registration (re-signing, ML-BOM annotation, registry enrollment). Excludes allowlist enforcement covered by SAFE-K8S-0905-013.

## Enforcement

- Primary Enforcement Point: Quarantine namespace with isolated scanning pipeline.
- Implementation Surface: Quarantine Kubernetes namespace with resource limits and restricted egress NetworkPolicy, automated scanning pipeline (pickle opcode analysis, format verification, known malicious pattern matching, embedded file detection), sandboxed container execution with syscall monitoring (seccomp, Falco), behavioral validation against standard evaluation datasets, provenance verification (GPG commit signatures, model card review, organization verification), internal re-signing using organizational Sigstore infrastructure, and ML-BOM annotation recording external origin and scan results.
- Owner Primary: Security Engineering
- Owner Supporting: ML Platform Engineering; ML Engineering
