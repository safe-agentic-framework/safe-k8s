# SAFE-K8S-0703-006 - Node affinity, taints, and topology-aware scheduling for AI workload isolation

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.3 - Resource Governance and Priority
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must configure node affinity rules, taints, and tolerations to isolate AI workloads from non-AI services, and must implement topology-aware scheduling for GPU locality (NVLink, NVSwitch) balanced against blast-radius containment.

## Objective

Prevent AI workloads from degrading non-AI services and optimize GPU scheduling for performance while maintaining isolation boundaries through Kubernetes scheduling controls.

## Rationale

AI training workloads consume massive CPU, memory, and network bandwidth, causing noisy-neighbor effects that degrade co-located non-AI services. Without node-level isolation through taints and tolerations, AI workloads can be scheduled on nodes running latency-sensitive services. Without topology-aware scheduling, distributed training jobs may be placed on GPUs without high-speed interconnects (NVLink), severely degrading performance and wasting expensive GPU time. However, concentrating all GPU workloads on a single failure domain increases blast radius. Host-level cgroup v2 and NUMA controls prevent resource contention at the kernel level.

## Scope

Applies to all Kubernetes clusters running mixed AI and non-AI workloads. Node taints must designate GPU nodes for AI workloads only. Topology-aware scheduling must consider GPU interconnect topology (NVLink, NVSwitch) for distributed training while spreading inference replicas across failure domains. Host-level isolation (cgroup v2, CPU pinning, NUMA) is required for nodes hosting both training and inference workloads. Dedicated single-purpose clusters may have simplified configurations.

## Enforcement

- Primary Enforcement Point: Node taint and toleration configuration, pod affinity and anti-affinity rules, topology spread constraints, and scheduler plugin configuration for GPU topology awareness.
- Implementation Surface: Kubernetes node taints and tolerations (nvidia.com/gpu=present:NoSchedule), pod spec.affinity.nodeAffinity with nodeSelectorTerms matching GPU product labels (nvidia.com/gpu.product), Topology Manager policies (single-numa-node, best-effort), node labels indicating GPU topology (NVLink, NVSwitch connectivity), and topology spread constraints for inference replica distribution across failure domains.
- Owner Primary: Platform Engineering
- Owner Supporting: ML Engineering; SRE
