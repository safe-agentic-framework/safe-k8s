# SAFE-K8S-1007-001 - Kubernetes and AI infrastructure upgrade planning

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.7 - Cluster Lifecycle and Asset Inventory
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must maintain documented upgrade plans for Kubernetes clusters that enforce version skew policy compliance, component-by-component upgrade sequencing, and tested rollback procedures, and must extend upgrade governance to AI infrastructure components by coordinating updates across GPU drivers, CUDA toolkits, AI operators, and model serving frameworks to prevent compatibility failures and security regressions.

## Objective

Ensure that Kubernetes and AI infrastructure upgrades are executed safely with maintained version compatibility, tested rollback paths, and coordinated AI component updates that do not introduce security vulnerabilities or service disruptions.

## Rationale

Kubernetes version skew violations and uncoordinated AI component updates create unpredictable cluster behavior and security gaps. GPU driver updates that are incompatible with the installed CUDA toolkit can crash AI workloads, and Kubernetes upgrades that skip versions can break AI operator CRDs. Without documented upgrade plans with rollback procedures, failed upgrades can leave clusters in a partially upgraded state that is both insecure and difficult to recover from.

## Scope

Applies to all Kubernetes clusters hosting AI workloads. Covers Kubernetes control plane and node upgrades, as well as AI-specific infrastructure components (GPU drivers, CUDA toolkits, device plugins, AI operators, serving frameworks). Managed Kubernetes services must document provider-managed upgrade coordination and customer-responsible AI component updates.

## Enforcement

- Primary Enforcement Point: Cluster lifecycle management tooling, change management approval process, and upgrade testing pipeline.
- Implementation Surface: Kubernetes version skew policy compliance verification, component-by-component upgrade sequencing documentation (control plane, node components, add-ons, admission controllers, operators), AI infrastructure compatibility matrix tracking (GPU Driver, CUDA Toolkit, cuDNN, PyTorch/TensorFlow, NVIDIA Device Plugin, DCGM Exporter, Triton/vLLM/TorchServe, MIG Manager version dependencies), non-production GPU node test environment for upgrade validation, rollback procedures per component (GPU driver rollback via node drain and reinstallation, operator rollback via CRD version management), and upgrade scheduling coordinated with training job idle windows and checkpoint/resume capability.
- Owner Primary: Platform Engineering
- Owner Supporting: AI/ML Engineering; Security Engineering; SRE
