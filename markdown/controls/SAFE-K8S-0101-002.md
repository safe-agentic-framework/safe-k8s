# SAFE-K8S-0101-002 - API server TLS configuration and certificate rotation

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.1 - Kubernetes API Server Security
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must enforce TLS for all API server connections, including client-facing front-end traffic and back-end component communication, by using TLS 1.2 or later with organization-approved cipher suites, certificates issued by a trusted CA, and automated certificate rotation before expiry.

## Objective

Protect API server communications from eavesdropping, tampering, and impersonation by enforcing TLS on all connection paths and ensuring certificates remain valid through automated rotation.

## Rationale

The API server communicates with kubelets, etcd, aggregated API servers, webhook backends, and external clients. Unencrypted or weakly encrypted connections expose cluster state and credentials to network-level attackers. Expired or improperly validated certificates create opportunities for man-in-the-middle attacks.

## Scope

Applies to all API server communication paths: client to API server (front-end), API server to etcd, API server to kubelet, API server to aggregated API servers, and webhook backends. For managed Kubernetes services, applies to customer-configurable TLS settings and to verification that provider-managed connections meet organizational TLS requirements.

## Enforcement

- Primary Enforcement Point: API Server Configuration
- Implementation Surface: Cluster provisioning templates, certificate management infrastructure (cert-manager, cloud provider CA), and periodic TLS configuration assessment.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; PKI/Certificate Management
