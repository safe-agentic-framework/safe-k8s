# SAFE-K8S-0704-009 - Cloud IAM to Kubernetes RBAC entitlement mapping governance

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must establish and maintain a documented mapping between cloud provider IAM identities and Kubernetes RBAC bindings for managed Kubernetes services, and must review that mapping whenever cloud IAM roles or Kubernetes RBAC bindings change so intended Kubernetes permissions remain explicit and traceable.

## Objective

Ensure cloud IAM identities are intentionally translated into Kubernetes permissions through an explicit, reviewable entitlement mapping rather than implicit or undocumented access paths.

## Rationale

Managed Kubernetes environments bridge cloud IAM and Kubernetes RBAC, creating a two-layer authorization model. Without an explicit entitlement mapping, teams cannot reliably determine which cloud identities resolve to which Kubernetes permissions, and access drift accumulates as IAM roles, access entries, and bindings evolve independently.

## Scope

Applies to managed Kubernetes services and any self-managed clusters using cloud IAM or external cloud identity for Kubernetes authentication. The mapping must cover administrative users, automation identities, CI/CD roles, monitoring roles, and any other cloud identity granted Kubernetes access.

## Enforcement

- Primary Enforcement Point: Cloud IAM identity definitions, Kubernetes authentication mappings, and RBAC binding review processes.
- Implementation Surface: EKS access entries or aws-auth mappings, GKE IAM-to-cluster role bindings, AKS Entra ID or equivalent identity integration, documented mapping matrices, and recurring review workflows comparing cloud identities to Kubernetes RBAC grants.
- Owner Primary: Cloud Platform Engineering
- Owner Supporting: Security Engineering; Platform Engineering
