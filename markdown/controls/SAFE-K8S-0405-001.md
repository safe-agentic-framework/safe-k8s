# SAFE-K8S-0405-001 - API impersonation restriction and auditing

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.5 - Identity Abuse Detection and Mitigation
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must restrict the Kubernetes API impersonation capability (impersonate verb on users, groups, and serviceaccounts resources) to explicitly authorized subjects only, and must audit all impersonation events through API server audit logging with alerts on unexpected impersonation usage.

## Objective

Prevent unauthorized identity assumption through the Kubernetes impersonation API, which allows a caller to act as another user, group, or service account, and ensure all legitimate impersonation usage is traceable.

## Rationale

The Kubernetes impersonation feature allows any subject with the impersonate verb to send API requests as any other user, group, or service account, effectively bypassing all RBAC restrictions of the impersonated identity. This is a powerful privilege escalation vector: an attacker with impersonation rights can act as cluster-admin or any service account without possessing their credentials. Unrestricted impersonation undermines the entire RBAC model.

## Scope

Applies to all clusters. Covers RBAC rules granting the impersonate verb on users, groups, serviceaccounts, userextras, and uid resources. Both human user impersonation and service account impersonation are in scope. Legitimate uses (CI/CD systems, multi-tenant controllers) must be documented with scoped impersonation targets.

## Enforcement

- Primary Enforcement Point: RBAC policies restricting the impersonate verb to explicitly approved subjects with scoped resourceNames, and API server audit policy configured to log all impersonation events at the RequestResponse level.
- Implementation Surface: RBAC ClusterRoles and Roles restricting the impersonate verb on users, groups, serviceaccounts, userextras, and uid resources with scoped resourceNames, API server audit policy configured to log impersonation events at RequestResponse level (impersonatedUser field detection), SIEM detection rules alerting on unexpected impersonation usage, and kubectl auth can-i impersonate verification tooling.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Security Operations
