# SAFE-K8S-1005-003 - Forensic evidence collection for Kubernetes AI incidents

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.5 - Incident Response for Kubernetes
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must establish forensic evidence collection procedures for Kubernetes incidents, including container filesystem snapshots, audit log preservation, and network flow capture, with documented chain-of-custody procedures, and must extend forensic collection to AI-specific evidence including GPU allocation logs, model access audit trails, and training data provenance records.

## Objective

Preserve complete, tamper-evident forensic evidence from Kubernetes AI infrastructure incidents to support root-cause analysis, legal proceedings, and post-incident improvements.

## Rationale

Container environments are ephemeral. Crashed or evicted pods lose their filesystem state, and network flows are not persisted by default. AI workloads produce additional volatile evidence, including GPU allocation state, model version checksums, and training data lineage, that is lost without proactive collection. Without established forensic procedures and chain-of-custody documentation, evidence may be inadmissible in legal proceedings or insufficient for determining whether a model was tampered with during an infrastructure compromise.

## Scope

Applies to all Kubernetes clusters hosting AI workloads where forensic readiness is required by policy or regulation. Covers container-level, node-level, and network-level evidence collection. AI-specific evidence collection must cover GPU allocation, model access, and training data provenance. Chain-of-custody procedures must meet organizational legal requirements.

## Enforcement

- Primary Enforcement Point: Forensic toolkit deployment, evidence collection automation scripts, and chain-of-custody documentation workflow.
- Implementation Surface: kubectl debug with ephemeral containers for live container filesystem capture, container runtime snapshot tools (containerd ctr snapshot), kube-apiserver audit log export to immutable storage, CNI flow logging (Calico, Cilium) and service mesh sidecar access log export, etcd snapshot tooling (etcdctl snapshot save), kubectl get -o yaml and kubectl describe for pod metadata preservation, DCGM telemetry export and nvidia-smi output capture for GPU allocation evidence, model CRD audit log filtering and registry access log collection, pipeline execution log and data source access log preservation for training data provenance, model artifact snapshots from PersistentVolumes or model registries, checkpoint history with timestamps and SHA-256 hashes, chain-of-custody documentation templates, and append-only evidence repository storage.
- Owner Primary: Security Operations
- Owner Supporting: Digital Forensics; AI/ML Engineering; Legal
