# SAFE-K8S-0905-038 - Model registry RBAC and workload pull authorization scoping

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.5 - AI Supply Chain and Model Lifecycle
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Model registries must enforce role-based access controls for publish, pull, modify, and delete operations and must scope workload and service-account access to approved repositories, projects, and environments. Human and machine identities must receive only the registry permissions required for their approved use cases.

## Objective

Prevent unauthorized publication, retrieval, and modification of model artifacts by enforcing scoped registry authorization for human and workload identities.

## Rationale

The model registry is the operational source of truth for deployable model artifacts. If registry roles or workload pull permissions are overly broad, an attacker or over-privileged workload can publish malicious models, retrieve restricted weights, or bypass environment boundaries.

## Scope

Applies to all model registry deployments on Kubernetes, including Harbor, MLflow Model Registry, and on-premises Hugging Face Hub instances. Covers project-scoped roles, OIDC or IAM integration, and workload pull permissions scoped to approved repositories and environments.

## Enforcement

- Primary Enforcement Point: Model registry authorization policy and workload identity scoping.
- Implementation Surface: Project-scoped registry roles, OIDC and cloud IAM integration, registry robot or service-account credentials, per-environment repository access rules, and pull-only permissions for inference workloads.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Security Engineering; Platform Engineering
