# SAFE-K8S-0102-002 - etcd access controls and network isolation

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must enforce client certificate authentication for all etcd client connections, enable mutual TLS for etcd peer communication, and isolate etcd at the network level so that only authorized control-plane components can reach etcd endpoints.

## Objective

Prevent unauthorized access to etcd by enforcing certificate-based authentication on all client and peer connections and restricting network reachability to control-plane components only.

## Rationale

etcd holds the authoritative cluster state; unrestricted access allows an attacker to read, modify, or delete any Kubernetes object without passing through API server authorization. Without client certificate authentication, any network-reachable process can connect. Without peer TLS, inter-node etcd traffic can be intercepted or spoofed. Without network isolation, worker nodes and workloads can reach etcd directly, bypassing all API server controls.

## Scope

Applies to all production Kubernetes clusters. Covers etcd client authentication (API server to etcd), etcd peer authentication (etcd member to member), and network-level controls (firewall rules, security groups, network policies) restricting etcd port access. For managed Kubernetes services, applies to verifying that etcd is not exposed to customer workloads and that provider-managed etcd meets TLS requirements.

## Enforcement

- Primary Enforcement Point: Node / OS
- Implementation Surface: etcd configuration flags (--client-cert-auth, --peer-client-cert-auth, --cert-file, --peer-cert-file), network firewall or security group rules, and periodic configuration and network reachability assessment.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Network Engineering
