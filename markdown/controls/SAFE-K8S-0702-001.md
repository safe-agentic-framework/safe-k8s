# SAFE-K8S-0702-001 - Namespace isolation model for multi-tenant AI clusters

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.2 - Namespace Isolation and Multi-Tenancy
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must define and enforce a namespace isolation model for multi-tenant clusters hosting AI workloads, must document the isolation properties and limitations of namespaces as a security boundary, and must implement supplementary controls (network policies, RBAC, admission control) to achieve the required isolation level between AI and non-AI workloads.

## Objective

Establish clear tenant isolation boundaries in clusters that co-host AI and non-AI workloads, preventing cross-tenant data leakage and resource interference.

## Rationale

Kubernetes namespaces provide logical separation but do not enforce strong security isolation by default. Without supplementary controls, workloads in one namespace can access services, storage, and network endpoints in other namespaces. In mixed AI/non-AI clusters, GPU-intensive training jobs may interfere with production inference services, and sensitive training data in one tenant's namespace may be accessible to another tenant's pods. Relying on namespace isolation alone creates a false sense of security.

## Scope

Applies to all multi-tenant Kubernetes clusters where multiple teams, projects, or workload types share infrastructure. Particularly critical for clusters hosting both AI workloads (training, inference) and non-AI services. Organizations must select and document their isolation model (namespace-per-team, namespace-per-workload-type, or hybrid) and justify it relative to their threat model.

## Enforcement

- Primary Enforcement Point: Namespace configuration, NetworkPolicy objects, RBAC bindings scoped to namespaces, admission control policies preventing cross-namespace resource references, and documentation of the chosen isolation model.
- Implementation Surface: Kubernetes Namespace resource definitions, namespace isolation model documentation (per-team, per-workload-type, per-project), Hierarchical Namespace Controller (HNC) for delegated namespace administration with policy inheritance, vCluster for virtual API-server-level tenant isolation, default-deny NetworkPolicy per namespace, ResourceQuota and LimitRange objects, namespace-scoped RBAC RoleBindings, admission control policies (Kyverno, OPA/Gatekeeper) for image sources and security contexts, and Pod Security Admission (PSA) enforce mode configuration.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; ML Engineering
