# SAFE-K8S-0402-017 - Projected token lifetime bounds and long-lived token exception governance

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.2 - Service Accounts and Workload Identity
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes clusters must enforce time-bounded projected service account token lifetimes, must track any remaining legacy or manually created long-lived service account token use toward zero, and must require documented, re-reviewed exceptions for any non-expiring or extended-lifetime token use.

## Objective

Reduce the credential-theft window for workload identities by constraining token lifetime and governing any exceptional long-lived token use until it is eliminated.

## Rationale

Audience scoping alone is insufficient if workload tokens remain valid indefinitely. Time-bounded token lifetimes reduce replay value after theft, and explicit tracking of remaining legacy or manually created long-lived tokens prevents insecure exceptions from becoming permanent hidden risk.

## Scope

Applies to all workload service account token use, including projected tokens with expirationSeconds settings and any remaining exceptional long-lived tokens created for constrained compatibility scenarios. Legacy token usage must be tracked and reviewed until eliminated.

## Enforcement

- Primary Enforcement Point: Token lifetime policy enforcement, monitoring, and exception governance for workload service account credentials.
- Implementation Surface: Projected token expirationSeconds settings, monitoring for legacy token secret presence or usage, documented exception records for extended-lifetime tokens, and audit evidence showing TokenRequest issuance and declining long-lived token dependence.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development
