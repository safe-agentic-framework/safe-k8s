# SAFE-K8S-0704-012 - Pod identity mechanisms replacing node-level instance profiles

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Foundational
- Classification: baseline
- Superseded By: SAFE-K8S-0704-019

## Statement

Organizations must configure per-workload cloud identity mechanisms and must not rely on node-level instance profiles to supply cloud API credentials to pods.

## Objective

Bind cloud API permissions to individual workloads instead of node-scoped credentials so a compromised pod cannot inherit cloud access intended for other workloads on the same node.

## Rationale

Node-level instance profiles attach cloud permissions to the underlying node and expose a broad lateral-movement opportunity when workloads can obtain those credentials through metadata access. Per-pod identity mechanisms narrow credential scope to the specific service account or workload identity binding, making cloud authorization independently auditable from metadata endpoint protections.

## Scope

Applies to all pods that require cloud API access on managed Kubernetes platforms. Covers IRSA, Pod Identity, Workload Identity Federation, Entra Workload ID, and restriction of node instance profiles to node bootstrap or infrastructure-only permissions.

## Enforcement

- Primary Enforcement Point: Per-Workload Cloud Identity Binding
- Implementation Surface: OIDC federation setup, workload identity bindings, service account annotations, pod identity admission/webhook configuration, and node instance profile restriction.
- Owner Primary: Cloud Platform Engineering
- Owner Supporting: Platform Engineering; Security Engineering
