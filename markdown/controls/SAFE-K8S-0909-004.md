# SAFE-K8S-0909-004 - Retrieval integrity and context poisoning defense

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.9 - RAG Infrastructure Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

RAG retrieval services must implement integrity checks on retrieved context before passing it to the LLM, must enforce relevance thresholds to prevent low-relevance or adversarial retrievals from entering the prompt, and must detect and mitigate context poisoning attacks where an adversary manipulates the vector index to influence LLM responses.

## Objective

Detect and prevent context poisoning attacks that manipulate LLM responses by controlling what content is retrieved from the vector index, and ensure that only high-relevance, authorized content reaches the LLM prompt.

## Rationale

Context poisoning is the defining threat for RAG systems. An attacker who can insert or modify documents in the knowledge base controls what context the LLM receives. Unlike model poisoning, context poisoning requires only write access to a document source or the vector index itself. The impact is immediate: the next query that retrieves the poisoned content receives a manipulated response. Low-similarity results are more likely to be adversarial injections designed to be retrieved for specific queries.

## Scope

Applies to all RAG retrieval services on Kubernetes. Covers similarity threshold enforcement, top-K limiting, index change monitoring, retrieval audit logging, and context poisoning detection. Excludes model-level defenses against adversarial inputs (see KA 9.3).

## Enforcement

- Primary Enforcement Point: Retrieval service similarity threshold configuration, index monitoring and alerting infrastructure, and retrieval audit logging pipeline.
- Implementation Surface: Retrieval service similarity threshold configuration, top-K result limiting, vector index change monitoring and alerting for unauthorized insertions or modifications, index snapshot and diff monitoring comparing current state against known-good baselines, retrieval audit logging (retrieved chunks, similarity scores, prompt assembly), source attribution in LLM responses, redundant retrieval from multiple independent indexes, and human-in-the-loop review for recently ingested document responses.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Security Engineering; Security Operations
