# SAFE-K8S-0603-002 - Build attestation taxonomy per NIST SP 800-204D

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.3 - Attestation, Provenance, and Cryptographic Assurance
- Maturity: Advanced
- Classification: baseline

## Statement

Build pipelines must generate attestations across all four NIST SP 800-204D categories: environment attestation documenting the build platform inventory, process attestation recording build tools and transformations applied, materials attestation capturing all source inputs and dependencies consumed, and artifacts attestation identifying build outputs and intermediate products. Each attestation category must be independently signed and verifiable.

## Objective

Ensure comprehensive build provenance by generating structured attestations that cover every dimension of the build process as defined by NIST SP 800-204D, enabling granular supply chain verification.

## Rationale

A single monolithic build attestation obscures which aspect of the build was verified. Without environment attestation, a compromised build runner goes undetected. Without materials attestation, dependency substitution attacks are invisible. Without process attestation, unauthorized build step modifications evade review. The four-category taxonomy ensures each attack vector has a corresponding verification checkpoint.

## Scope

Applies to all CI/CD pipelines that produce container images or deployable artifacts for production clusters. Organizations using managed build services must verify that the provider generates equivalent attestation categories or implement compensating attestation generation.

## Enforcement

- Primary Enforcement Point: CI/CD pipeline attestation generation stages and policy engine evaluation at admission time.
- Implementation Surface: NIST SP 800-204D four-category attestation taxonomy generation in CI/CD pipelines (environment attestation for build platform inventory, process attestation for build tools and transformations, materials attestation for source inputs and dependencies, artifacts attestation for build outputs), in-toto statement format with per-category independent signatures, CI runner image digest and node attestation capture, Dockerfile content hashing, dependency lock file hash recording, and policy engine rules verifying presence of all four attestation types at admission.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: Security Engineering; Platform Engineering
