# SAFE-K8S-0501-005 - AI workload network segmentation

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.1 - Network Policies
- Maturity: Foundational
- Classification: ai-specific

## Statement

AI workloads must be microsegmented so that training, inference, pipeline orchestration, and notebook environments reside in separate network segments with explicit policy-controlled communication paths, and model download paths must be isolated from training data paths to prevent cross-contamination.

## Objective

Prevent cross-contamination between AI workload types by enforcing network-layer isolation between workload segments and separating model supply-chain paths from training data paths.

## Rationale

AI workloads handle high-value assets including training data, model weights, and inference results. Without segmentation, a compromised notebook environment can directly access production inference endpoints or training data stores. Mixing model download paths with training data paths creates risk of supply-chain attacks contaminating training pipelines.

## Scope

Applies to all Kubernetes clusters hosting AI/ML workloads including training jobs, inference services, pipeline controllers (Kubeflow, Argo), and interactive notebook environments. Includes both GPU and CPU-based AI workloads.

## Enforcement

- Primary Enforcement Point: Kubernetes NetworkPolicy and CNI-specific policies segmenting AI workload namespaces.
- Implementation Surface: Kubernetes NetworkPolicy objects segmenting AI workload namespaces (ml-training-*, ml-inference-*, ml-pipeline, ml-notebooks), CiliumClusterwideNetworkPolicy with workload-type label selectors (training, inference, pipeline, notebook), separate egress rules isolating model download paths from training data storage paths, and connectivity verification scripts testing cross-segment isolation.
- Owner Primary: Network Engineering
- Owner Supporting: ML Platform Engineering
